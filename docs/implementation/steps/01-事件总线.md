# 01-事件总线

## 1. 目标

定义 FDE Workstation 的统一事件入口和事件基础设施。事件总线只做事件接入、标准化、路由分发、可靠投递、事件归档和消费端 SDK，不承载 Pipeline、Diagnosis、Collaboration 的业务规则。

本文档按目标架构定义，不按临时最小实现收缩范围。当前事件总线使用 Redis Streams 承担可靠事件路由、消费组、ack、回放和死信；Redis 同时承担幂等去重和短期状态缓存。PostgreSQL 属于事件归档、审计和回放查询目标架构的一部分，代码实现可以分批完成，但接口、数据结构和职责边界必须先定住。

核心链路：

```text
事件接入层 -> 标准化层
  ├─ 路由分发层 -> 可靠投递层 -> Agent 消费端 SDK
  └─ 事件归档层（异步写入，不阻塞事件投递）
```

## 2. 不做什么

```text
不判断是否应该部署
不判断是否应该通知谁
不判断故障根因
不保存业务 Agent 的长期状态机
不把 PostgreSQL 当作业务状态库
不把 Redis 当作永久存储
不在事件 data 中保存大日志或真实密钥
不让上层 Agent 直接适配 GitLab、Tekton、ArgoCD、K8s、Feishu 的原始 payload
```

## 3. 技术选型

| 类别 | 选型 | 说明 |
| --- | --- | --- |
| 开发语言 | TypeScript / Node.js | 与 Agent Runtime 保持技术栈一致 |
| 服务框架 | NestJS | 模块化、依赖注入、适合基础设施服务 |
| HTTP 入口 | NestJS Controller 或 Fastify adapter | 承接 webhook、内部上报和健康检查 |
| 消息中间件 | Redis Streams | stream、consumer group、ack、pending、回放能力满足当前内部 DevOps 规模 |
| 幂等存储 | Redis | 入口去重、消费去重、短期投递状态 |
| 事件归档 | PostgreSQL | 全量事件归档、审计、回放、排障查询 |
| 事件协议 | CloudEvents 1.0 | 避免自定义协议扩散 |
| Schema | JSON Schema | 对事件信封、data、投递记录做结构校验 |

## 4. 总体架构

```text
external systems
  ├─ GitLab Webhook
  ├─ Tekton HTTP Report Task
  ├─ Tekton / K8s Informer
  ├─ ArgoCD Webhook
  ├─ K8s Event Exporter
  └─ Feishu Callback
        ↓
event-ingress
        ↓
event-normalizer
        ├─ async archive -> EventArchiveRepository(PostgreSQL)
        ↓ publish
EventBroker(Redis Streams)
        ↓
agent queues
        ↓
EventSubscriber SDK
        ↓
Pipeline / Diagnosis / Collaboration
```

模块职责：

| 模块 | 职责 |
| --- | --- |
| `event-ingress` | 接收外部事件，做认证、签名校验、来源识别 |
| `event-normalizer` | 把异构 payload 转成 CloudEvents |
| `event-router` | 根据 `type` 发布到 Redis Stream |
| `event-delivery` | 管理重试、死信、消费确认 |
| `event-archive` | 异步写入 PostgreSQL，支持审计和回放 |
| `event-subscriber-sdk` | 给业务 Agent 提供订阅、ack、nack、幂等封装 |

## 5. 事件接入层

事件接入层负责接收多源事件，并完成来源校验。不同来源的信任方式不同，不能强行统一为同一种签名模型。

| 事件源 | 接入方式 | 校验方式 | 核心事件 |
| --- | --- | --- | --- |
| GitLab | HTTP Webhook | `X-Gitlab-Token` 或 GitLab Secret Token | MR 打开、MR 更新、评论、push、pipeline 状态 |
| Tekton | PipelineRun 末尾 HTTP 上报 Task | 内部 Token、服务账号、网络边界 | 构建开始、成功、失败、镜像构建完成 |
| Tekton | K8s Informer 监听 PipelineRun / TaskRun | Kubernetes RBAC、命名空间限制 | 构建状态兜底采集 |
| ArgoCD | Webhook | 签名 Token 或共享 Secret | 同步状态变化、应用健康度变化 |
| K8s | Event Exporter 或 Informer | Kubernetes RBAC、命名空间限制 | Pod 异常、节点异常、资源事件 |
| Feishu | Callback | verification token、encrypt key | 卡片点击、消息回复 |

入口规则：

```text
1. 对外 HTTP webhook 必须校验签名或 token。
2. 集群内部 informer / exporter 必须依赖 Kubernetes RBAC 和网络边界。
3. 所有入口必须生成或继承 correlation_id、trace_id、event_id。
4. 原始 payload 必须先脱敏，再进入标准化流程。
5. 大 payload 先写 artifact，事件 data 中只保留引用。
```

## 6. 标准化层

所有异构事件必须转换成 CloudEvents 1.0 风格信封。上层业务 Agent 只能消费标准事件，不直接依赖外部系统原始 payload。

### 6.1 CloudEvent 信封

```json
{
  "specversion": "1.0",
  "id": "evt-20260617-0001",
  "source": "gitlab",
  "type": "gitlab.mr.updated",
  "subject": "project/123/mr/45",
  "time": "2026-06-17T08:00:00Z",
  "datacontenttype": "application/json",
  "correlation_id": "corr-20260617-0001",
  "trace_id": "trace-20260617-0001",
  "run_id": "run-20260617-0001",
  "application": "api-gateway",
  "environment": "dev",
  "data": {},
  "metadata": {
    "upstream_id": "gitlab-project-123-mr-45-update-1",
    "adapter_version": "gitlab-webhook-v1"
  }
}
```

字段规则：

| 字段 | 规则 |
| --- | --- |
| `specversion` | 固定为 `1.0` |
| `id` | FDE 内部事件 ID，全局唯一 |
| `source` | 事件来源，使用系统名，例如 `gitlab`、`tekton`、`argocd`、`kubernetes`、`feishu` |
| `type` | 事件类型，使用 `系统.对象.动作` |
| `subject` | 事件主体，例如 `project/123/mr/45` |
| `time` | 事件发生时间，UTC ISO 8601 |
| `correlation_id` | 业务链路 ID |
| `trace_id` | 技术链路 ID |
| `run_id` | 执行实例 ID，可由入口或下游任务生成 |
| `application` | 业务应用名 |
| `environment` | `dev`、`test`、`prod` |
| `data` | 标准化后的业务数据 |
| `metadata` | 来源系统差异字段 |

### 6.2 命名约定

```text
事件 type 使用 lower dot notation：
系统.对象.动作

示例：
gitlab.mr.updated
tekton.pipelinerun.completed
argocd.application.degraded
kubernetes.pod.failed
collaboration.notification.sent
```

### 6.3 data 约束

```text
1. data 必须是标准化结构，不直接保存完整原始 payload。
2. 原始 payload 如需保留，写入 raw_event_payload artifact。
3. data 中的状态字段统一使用 status。
4. data 中的大日志必须转为 artifact_refs。
5. data 不允许包含 token、cookie、secret、password、private_key。
```

## 7. 路由分发层

事件总线使用 Redis Streams 进行发布订阅和可靠消费。

```text
stream:
- fde.events

routing_key:
- 等于 CloudEvent.type

consumer group:
- agent.pipeline
- agent.diagnosis
- agent.collaboration
- agent.audit
```

路由规则：

```text
1. 事件发布方只写入 stream，不关心具体消费者。
2. 每个 Agent 使用独立 consumer group，并在 SDK 内按 event type 过滤。
3. 新增 Agent 不需要修改事件总线发布逻辑。
4. 一个事件可以被多个 consumer group 消费。
5. Redis Streams 只负责事件投递和消费状态，不负责业务状态。
6. 同一个 consumer group 内如果存在多个订阅，SDK 必须先合并订阅再按 event type 分发，不能让某个订阅把不关心的事件直接 ack 掉。
```

典型订阅关系：

| 消费者 | 订阅事件 |
| --- | --- |
| Pipeline Agent | `tekton.pipelinerun.completed`、`gitlab.mr.merged`、`gitops.yaml.updated` |
| Pipeline Agent / MR Review 模块 | `gitlab.mr.created`、`gitlab.mr.updated`、`gitlab.mr.comment.created` |
| Diagnosis Agent | `argocd.application.degraded`、`kubernetes.pod.failed`、`pipeline.deployment.failed` |
| Collaboration Agent | `diagnosis.completed`、`collaboration.notification.timeout`、异常类事件 |
| Audit Consumer | `#` |

## 8. 可靠投递层

可靠投递层保证事件至少被投递一次，并把失败显式暴露出来。

### 8.1 投递语义

```text
投递语义：at-least-once
消费端必须幂等
不承诺 exactly-once
不允许静默丢事件
```

### 8.2 重试策略

```text
默认重试次数：3 到 5 次，由 consumer group policy 配置
退避策略：指数退避
重试对象：消费失败、暂时性上游不可用、可重试处理错误
不可重试错误：schema 不合法、权限失败、签名失败、payload 无法解析
```

### 8.3 死信队列

```text
dead letter stream:
- fde.events.dlq

dead letter consumer group:
- dlq.pipeline
- dlq.diagnosis
- dlq.collaboration
- dlq.audit
```

死信规则：

```text
1. 重试耗尽必须进入死信队列。
2. 死信事件必须写入 PostgreSQL。
3. 死信必须触发告警或协同通知。
4. 死信不能自动丢弃。
5. 死信事件必须保留原始 event_id、consumer_id、失败原因和投递次数。
```

## 9. 幂等策略

幂等分为入口幂等和消费幂等，不能只用一个全局 `event_id` 判断。

### 9.1 入口幂等

用于防止同一个外部事件重复进入总线。

```text
key:
ingress:{source}:{upstream_id}

value:
event_id

ttl:
按事件源配置，默认 7 天
```

如果外部事件没有稳定 ID，adapter 必须用稳定字段生成 `upstream_id`，例如项目 ID、MR ID、状态、更新时间组合。

### 9.2 消费幂等

用于防止同一个消费者重复处理同一事件。

```text
key:
consumer:{consumer_id}:{event_id}

value:
processed | processing | failed

ttl:
按消费者配置，默认 7 到 30 天
```

消费幂等规则：

```text
1. 一个事件被 Pipeline Agent 消费，不代表 Diagnosis Agent 已消费。
2. consumer_id 必须参与幂等 key。
3. handler 成功后才能标记 processed。
4. handler 失败时不得误标 processed。
```

## 10. 事件归档层

事件归档使用 PostgreSQL，职责是审计、排障、回放和统计，不是业务 Agent 状态机。

目标表：

```text
event_archive:
- id
- event_id
- source
- type
- subject
- correlation_id
- trace_id
- run_id
- application
- environment
- event_time
- received_at
- data_json
- metadata_json
- raw_payload_artifact_uri
- schema_version

event_delivery:
- id
- delivery_id
- event_id
- consumer_id
- queue_name
- status
- attempt_count
- first_attempt_at
- last_attempt_at
- next_retry_at
- error_json

dead_letter_event:
- id
- event_id
- consumer_id
- queue_name
- failed_at
- reason_code
- error_json
- raw_event_artifact_uri
```

归档规则：

```text
1. 标准化成功后的事件必须写入 event_archive。
2. 每次投递尝试必须写入或更新 event_delivery。
3. 死信必须写入 dead_letter_event。
4. 原始大 payload 写 artifact，表中保存引用。
5. 标准事件写入 Redis Streams 与写入 PostgreSQL 归档采用并行异步路径，归档慢不得阻塞事件投递。
6. 归档失败不能静默吞掉，必须记录 ARCHIVE_WRITE_FAILED，并把失败写入 artifact 或补偿队列。
7. 如果发布失败但归档成功，必须保留待重放标记，供后续人工或自动重放。
```

## 11. 消费端 SDK

业务 Agent 不直接操作 Redis 客户端或 stream 游标，应通过 EventSubscriber SDK 订阅事件。

SDK 目标能力：

```text
subscribe(event_types, handler, options)
ack(delivery_id)
nack(delivery_id, error)
publish(event)
publish_dead_letter(event, error)
mark_processed(consumer_id, event_id)
check_processed(consumer_id, event_id)
```

handler 形态：

```typescript
type EventHandler<TData> = (event: CloudEvent<TData>, context: DeliveryContext) => Promise<void>;
```

`DeliveryContext` 必须包含：

```text
delivery_id
stream_id
consumer_id
queue_name
attempt_count
received_at
trace_id
correlation_id
```

SDK 规则：

```text
1. SDK 在调用 handler 前检查消费幂等。
2. SDK 负责 ack / nack 的标准流程。
3. SDK 负责写 event_delivery。
4. SDK 不执行业务补偿逻辑。
5. handler 内部不得直接确认 Redis Stream 消息。
```

## 14. 与下一步衔接

`02-智能体运行时.md` 从 CloudEvent 中读取 `correlation_id`、`trace_id`、`run_id`、`application`、`environment` 和 `data`，生成 Agent task。

`04-合规检测雷达.md` 发布 `compliance.environment.scan.completed` 或 `compliance.environment.scan.failed`。

`05-Pipeline智能体核心.md` 消费 `tekton.pipelinerun.completed`，并发布 `gitops.yaml.updated`、`pipeline.build.completed` 或 `pipeline.deployment.failed`。

`09-诊断上下文构建器.md` 消费异常类事件，生成 `diagnosis.context.built`。

`13-协同通知路由.md` 消费 `diagnosis.completed`，生成 `collaboration.notification.requested`。

## 16. 风险与降级

| 风险 | 降级 |
| --- | --- |
| Redis 暂未部署 | `EventBroker` 和 `IdempotencyStore` 可临时使用本地 adapter，但接口必须保持发布订阅、ack 和幂等语义 |
| PostgreSQL 暂未部署 | `EventArchiveRepository` 可临时写 artifact，但归档记录结构不得变化 |
| 外部 webhook 未接通 | 使用 sample event 驱动后续模块 |
| 事件类型不完整 | 允许新增 type，但必须先改本文档和 schema |
| 事件体过大 | 大对象转 artifact，只在 data 中放引用 |
| 消费失败反复重试 | 指数退避后进入死信队列并告警 |
| 重复事件进入总线 | 入口幂等拦截 |
| 同一事件被同一消费者重复处理 | 消费幂等拦截 |

## 17. 验收补充

```text
事件总线必须支持 GitLab、Tekton、ArgoCD、K8s、Feishu 事件订阅。
事件总线必须能把标准事件投递给业务 Agent consumer group，并保留 dead letter stream 路径。
```

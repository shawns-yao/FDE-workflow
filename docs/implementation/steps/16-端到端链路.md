# 16-端到端链路

## 1. 目标

验证并固化完整链路：GitLab 提交和 MR 触发 Pipeline Agent，Tekton 构建和 YAML 更新推进 GitOps，ArgoCD 和 K8s 异常进入 Diagnosis Agent，诊断结果进入 Collaboration Agent，最终形成通知、进度追踪和日报。

## 2. 不做什么

```text
不引入新的业务智能体
不把所有能力压成一个脚本
不跳过事件、artifact 和 schema 契约
不在 prod 环境自动执行高风险修复
```

## 3. 上游输入

```text
gitlab.mr.created
gitlab.pipeline.completed
tekton.pipelinerun.completed
gitops.yaml.updated
argocd.application.sync.requested
argocd.application.synced
kubernetes.pod.failed
diagnosis.completed
collaboration.notification.sent
collaboration.progress.updated
```

## 4. 输出结果

```text
end-to-end-run-record.json
pipeline-agent-result.json
diagnosis-agent-result.json
collaboration-agent-result.json
daily-report.md
```

### 5.2 链路约束

```text
每个阶段必须产生 event_id
每个 AI 调用必须产生 artifact_uri
每个跨系统动作必须记录 target 和 result
每个失败必须有可重试或人工接管路径
```

## 6. 处理流程

```text
1. GitLab MR 触发 MR 评审智能体
2. GitLab Pipeline 或 Tekton 触发 Pipeline Agent 核心
3. 构建成功后更新 GitOps YAML 镜像 tag
4. YAML 治理智能体校验或修复配置
5. 推送配置仓库并触发 ArgoCD 同步
6. ArgoCD 和 K8s 异常事件进入诊断上下文构建器
7. 规则引擎、知识库匹配和 Claude API 按漏斗顺序生成诊断结果
8. 协同通知路由将诊断结果推送给责任人
9. 进度追踪识别响应、修复和升级状态
10. 日报生成器沉淀当天链路结果
```

## 8. 与下一步衔接

本步骤不是新的功能模块，而是实施前的总装检查。通过后才能进入代码落地阶段，顺序应从 `00-基础契约.md` 到 `03-飞书连接器.md` 先实现底座，再实现 `04-合规检测雷达.md`，随后实现 Pipeline、Diagnosis 和 Collaboration 的业务步骤。

## 9. 验证方式

```text
定向验证：用 mock 事件串起 GitLab -> Tekton -> YAML -> ArgoCD -> Diagnosis -> Feishu。
定向验证：每个阶段输出都能被下一个阶段读取。
定向验证：失败路径必须有 failed event 或 fallback artifact。
定向验证：prod 环境只允许建议、审批和通知，不允许自动高风险变更。
```

### 9.1 fixture 编排

```text
fixtures/
  e2e/
    runbook.yaml
    01-gitlab-mr-created.json
    02-gitlab-pipeline-completed.json
    03-tekton-pipelinerun-completed.json
    04-gitops-yaml-updated.json
    05-argocd-application-sync-requested.json
    06-argocd-application-degraded.json
    07-kubernetes-pod-failed.json
    08-diagnosis-completed.json
    09-collaboration-notification-sent.json
    10-collaboration-progress-updated.json
```

`runbook.yaml` 只描述事件顺序、输入文件、期望输出和失败注入点，不负责启动真实外部系统。端到端验证第一版使用 fixture runner 或人工顺序执行脚本，真实 GitLab / Tekton / ArgoCD / K8s / 飞书联调必须等 `04-合规检测雷达.md` 通过后再进行。

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 外部系统凭据未就绪 | 使用 mock connector 和 fixture artifact |
| 某个智能体不可用 | 保留确定性链路，跳过 AI 增强并记录 skipped |
| 事件链路断裂 | 通过 end-to-end-run-record 定位最后成功阶段 |
| prod 策略不明确 | 默认禁止自动修复和自动回滚 |

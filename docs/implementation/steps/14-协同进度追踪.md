# 14-协同进度追踪

## 1. 目标

实现 Collaboration Agent 的进度闭环：跟踪飞书通知后的已读、认领、回复、修复和超时状态，识别无效回复，并在需要时升级给更高优先级对象。

## 2. 不做什么

```text
不替代人工审批
不直接判断业务是否完全恢复
不修改 GitLab、ArgoCD 或 K8s 状态
不把“有人回复”视为“问题已解决”
```

## 3. 上游输入

```text
collaboration.notification.sent
feishu.card.action_clicked
feishu.message.replied
collaboration.notification.timeout
diagnosis.completed
```

## 4. 输出结果

```text
collaboration.progress.updated
collaboration.escalation.triggered
progress-record.json
reply-classification.json
```

## 5. 接口契约

### 5.1 进度状态

```json
{
  "notification_id": "ntf-001",
  "diagnosis_id": "diag-002",
  "status": "investigating",
  "actor": {
    "type": "user",
    "id": "ou_xxx"
  },
  "latest_reply": "我正在查看配置中心连接",
  "reply_effectiveness": "effective",
  "updated_at": "2026-06-17T10:30:00+08:00"
}
```

### 5.2 状态枚举

```text
unread
acknowledged
investigating
fixed
ineffective_reply
needs_escalation
escalated
```

## 6. 处理流程

```text
1. 读取 notification.sent 中的 notification_id 和 message_id
2. 接收飞书按钮回调或回复事件
3. 将按钮动作映射为进度状态
4. 对自然语言回复调用 Claude API 做有效性识别
5. 无回复、无效回复或超时未修复时触发升级策略
6. 输出 progress.updated 或 escalation.triggered
7. 保存 progress-record 和 reply-classification artifact
```

```text
agents/

```

## 8. 与下一步衔接

进度记录进入 `15-协同日报生成.md`。日报需要区分已通知、已响应、处理中、已修复、已升级和无效回复，不能只统计消息发送数量。

## 9. 验证方式

```text
定向验证：按钮 acknowledge 后状态变为 acknowledged。
定向验证：无效回复不能关闭事件。
定向验证：超时未处理必须输出 escalation.triggered。
定向验证：回复分类结果必须包含 evidence 和 confidence。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 飞书回调丢失 | 依赖 timeout 事件进入人工确认 |
| Claude API 不可用 | 只使用按钮动作和关键词规则判断回复 |
| 回复内容含糊 | 标记 ineffective_reply，要求明确处理动作 |
| 升级策略配置缺失 | 发送到默认运维值班群 |

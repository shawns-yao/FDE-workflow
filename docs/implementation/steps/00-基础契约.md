# 00-基础契约

## 1. 目标

建立所有后续步骤共享的基础契约，包括 ID、环境、时间、错误结构、Artifact 引用、Schema 命名、Fixture 编排、敏感信息脱敏、基础设施抽象和目录约定。

本文档不是临时实现说明，而是 FDE Workstation 的目标契约基线。即使某些能力在当前代码中尚未实现，也必须先定义接口边界，避免后续引入 Redis Streams、PostgreSQL、对象存储、审计和回放能力时重写上层 Agent。

后续事件总线、智能体运行时、Pipeline Agent、Diagnosis Agent、Collaboration Agent 都必须消费这些契约，不能各自重新定义字段和枚举。

IM 连接器、合规检测雷达和后续平台级基础设施也必须复用这些契约。

## 2. 不做什么

```text
不实现业务逻辑
不接入真实 GitLab / Tekton / ArgoCD / K8s / 飞书
不调用 Claude API
不把数据库当作业务逻辑前置条件
不把事件总线、运行时、业务 Agent 的状态结构混在一起
不把临时无数据库实现写成长期接口事实
```

## 3. 上游输入

本步骤没有业务上游。输入来自架构基线和后续目标能力：

```text
三类业务 Agent：
- Pipeline Agent
- Diagnosis Agent
- Collaboration Agent

基础设施目标：
- Redis Streams 事件路由、可靠投递、消费组和回放
- Redis 幂等去重和短期状态缓存
- PostgreSQL 事件归档、审计、回放和任务记录
- Artifact 本地目录或对象存储
- JSON Schema 契约校验
```

## 5. 通用字段契约

### 5.1 基础字段

```json
{
  "correlation_id": "corr-m3x5k9r2-a1b2c3d4e5f6g7h8",
  "trace_id": "trace-m3x5k9r2-b1c2d3e4f5a6b7c8",
  "run_id": "run-m3x5k9r2-c1d2e3f4a5b6c7d8",
  "application": "api-gateway",
  "environment": "dev",
  "created_at": "2026-06-17T08:00:00Z",
  "metadata": {}
}
```

字段含义：

| 字段 | 含义 | 生成方 | 是否必填 |
| --- | --- | --- | --- |
| `correlation_id` | 关联一次业务链路，从外部事件进入后贯穿所有 Agent | 事件总线或上游 adapter | 是 |
| `trace_id` | 技术链路追踪 ID，用于日志、指标、调用链 | 入口服务 | 是 |
| `run_id` | 一次执行实例 ID，用于 artifact、任务和审计归档 | 任务发起方 | 是 |
| `application` | 业务应用名，例如 `api-gateway` | 上游事件或 adapter | 是 |
| `environment` | 环境枚举 | 上游事件或 adapter | 是 |
| `created_at` | 创建时间，UTC ISO 8601 | 当前写入方 | 是 |
| `metadata` | 扩展字段 | 当前写入方 | 否 |

### 5.2 ID 规范

```text
event_id：事件唯一 ID，用于事件归档和入口幂等
correlation_id：业务链路 ID，贯穿事件、任务、诊断、通知
trace_id：技术链路 ID，用于日志和可观测性
run_id：一次执行实例 ID，用于 artifact 目录和任务记录
task_id：Agent Runtime 任务 ID
skill_id：Agent Skill 能力包 ID
scan_id：合规检测雷达扫描 ID
artifact_id：artifact 唯一 ID
diagnosis_id：诊断结果 ID
notification_id：通知 ID
consumer_id：事件消费者 ID
delivery_id：一次投递尝试 ID
```

ID 生成规则：

```text
1. 外部系统已有稳定 ID 时，应保留在 upstream_id 中。
2. FDE 内部 ID 必须全局唯一，使用时间有序 ID（类 UUID v7 结构）。
   格式: {prefix}-{timestamp_base36}-{random_hex}
   示例: evt-m3x5k9r2-a1b2c3d4e5f6g7h8
   特性：按时间排序（毫秒精度）、全局唯一（128位随机熵）、可读性好（base36编码）
3. 不允许把数据库自增 ID 暴露为跨模块主键。
4. 幂等主键必须由稳定业务字段组成，不能依赖随机 ID。
```

### 5.3 环境枚举

```text
dev
test
prod
```

环境策略：

| 环境 | 写入策略 | 自动化策略 |
| --- | --- | --- |
| `dev` | 允许受限自动写入 | 可自动生成 patch 和提交建议 |
| `test` | 允许受限自动写入 | 必须生成 diff artifact |
| `prod` | 禁止自动写入业务仓库 | 只能生成建议、patch artifact 和人工确认步骤 |

### 5.4 时间格式

```text
统一使用 UTC ISO 8601，例如 2026-06-17T08:00:00Z
禁止使用本地时区字符串作为持久化字段
禁止使用自然语言时间作为 schema 字段
```

允许额外记录本地展示时间，但必须放入 `metadata.display_time`，不能替代标准时间字段。

### 5.5 命名规范

```text
JSON 字段统一使用 snake_case
事件 type 使用 lower dot notation，例如 gitlab.mr.updated
文件名使用 kebab-case
目录名使用 kebab-case 或既有项目约定
枚举值使用 lower_snake_case
错误 code 使用 UPPER_SNAKE_CASE
Agent Skill 名称使用 kebab-case，例如 fde-pipeline
MCP server 名称使用 lower_snake_case 或 kebab-case，工具暴露给 Runtime 时统一为 mcp__<server>__<tool>
```

### 5.6 Agent Skill 与 MCP 工具契约

FDE 参考 Agentic Skills 的能力组织方式，但不把 Skill 和 MCP 混为一层。

```text
Agent Skill：
- 描述业务能力、触发条件、上下文材料、Prompt 约束、输入输出 schema 和推荐工具清单
- 可以包含 SKILL.md、参考文档、prompt 模板、fixture 和 schema 引用
- 不保存密钥，不直接调用外部系统，不绕过 Runtime 权限

MCP 工具：
- 描述可执行工具和参数 schema
- 负责 GitLab、Tekton、ArgoCD、K8s、飞书、知识库等外部系统能力
- 调用失败必须映射为 ErrorObject
- 必须受 permission_profile、allowed_tools、环境策略和审计约束
```

两者关系：

```text
业务 Agent 选择 Skill
  -> Skill 提供任务知识、prompt、schema 和建议工具
    -> Runtime 按 permission_profile / allowed_tools 装配工具池
      -> MCP provider 暴露外部系统工具
        -> 工具调用生成 tool trace 和 artifact
```

Skill 只能声明“希望使用哪些工具”，不能授予权限。最终可见工具仍以 Runtime 权限裁剪结果为准。

最小 Skill 描述示例：

```json
{
  "skill_id": "skill-fde-pipeline",
  "name": "fde-pipeline",
  "business_task_types": ["yaml_governance", "build_fix"],
  "runtime_capabilities": ["code_task", "repair_task"],
  "required_artifacts": ["diff", "build_log"],
  "recommended_tools": ["read_file", "list_files", "mcp__tekton__get_task_logs"],
  "output_schema_refs": ["schemas/pipeline/yaml-governance-result.schema.json"]
}
```

## 6. 错误结构

```json
{
  "code": "UPSTREAM_UNAVAILABLE",
  "message": "脱敏后的可读错误",
  "retryable": true,
  "severity": "warning",
  "details": {
    "system": "argocd",
    "operation": "get_application_status"
  },
  "cause_ref": "artifacts/runs/run-20260617-0001/errors/raw-error.json"
}
```

字段约束：

| 字段 | 含义 |
| --- | --- |
| `code` | 稳定错误码，供下游判断 |
| `message` | 脱敏后的可读错误 |
| `retryable` | 是否允许自动重试 |
| `severity` | `info`、`warning`、`error`、`critical` |
| `details` | 脱敏结构化上下文 |
| `cause_ref` | 原始错误 artifact 引用，可选 |

常用错误码：

## 7. Artifact 引用

```json
{
  "artifact_id": "artifact-20260617-0001",
  "artifact_uri": "artifacts/runs/run-20260617-0001/yaml-audit-report.md",
  "artifact_type": "yaml_audit_report",
  "content_type": "text/markdown",
  "sha256": "optional-content-hash",
  "size_bytes": 12345,
  "created_at": "2026-06-17T08:00:00Z",
  "excerpt": "脱敏摘要"
}
```

Artifact 规则：

```text
1. 大日志、大 payload、模型原始输出、diff、patch、trace 都必须写入 artifact。
2. 事件和任务结果中只保存 artifact 引用，不内联大对象。
3. artifact_uri 当前可以是本地相对路径，后续可迁移到对象存储。
4. artifact_type 必须是稳定枚举，不能随意写自然语言。
5. excerpt 必须脱敏。
6. 默认目录为 artifacts/runs/<run_id>/；领域长期报告可使用 artifacts/<domain>/<domain_id>/，例如 artifacts/compliance/<scan_id>/。事件和任务结果必须通过 artifact_ref 或 artifact_refs 引用，不能依赖调用方猜测目录。
```

常见 artifact 类型：

```text
raw_event_payload
normalized_event
tool_trace
agent_task_input
agent_task_result
diagnosis_context
diagnosis_result
notification_card
yaml_audit_report
code_review_report
build_repair_report
environment_check_report
patch
diff
dead_letter_event
delivery_attempt_log
```

## 8. Metadata 扩展规则

`metadata` 用于兼容外部系统差异和后续扩展，但不能绕过 schema 契约。

```text
允许：
- adapter_version
- upstream_system
- upstream_id
- namespace
- project_id
- repository
- branch
- commit_sha
- actor
- display_time
- pending_checks

禁止：
- token
- cookie
- password
- secret
- private_key
- access_key
- authorization
- 原始大日志
- 未脱敏异常堆栈
```

新增稳定字段时，不能长期塞在 `metadata` 中。若字段被两个以上模块依赖，应提升为正式 schema 字段。

## 9. 敏感信息脱敏

必须脱敏的字段名片段：

```text
token
secret
password
passwd
authorization
cookie
private_key
access_key
refresh_token
client_secret
webhook_secret
app_secret
encrypt_key
```

脱敏规则：

```text
1. 日志、artifact 摘要、错误 message、模型输入都必须脱敏。
2. 原始 payload 如需归档，只能写入受控 artifact，并标记访问级别。
3. 默认不把密钥类字段传给模型。
4. 脱敏后保留字段名，便于排障，但值统一替换为 [REDACTED]。
```

## 10. 基础设施抽象

基础设施可以分批实现，但接口必须从第一天存在。

| 抽象 | 目标实现 | 职责 |
| --- | --- | --- |
| `EventBroker` | Redis Streams | 事件发布、路由、订阅、消费组、ack 和死信 |
| `IdempotencyStore` | Redis | 入口去重、消费去重、短期状态缓存 |
| `EventArchiveRepository` | PostgreSQL | 事件归档、审计、回放、排障查询 |
| `ArtifactStore` | 本地目录或对象存储 | 保存大对象、报告、trace、patch |
| `SchemaRegistry` | JSON Schema 文件 | 校验事件、任务、结果 |
| `SecretProvider` | 环境变量或密钥系统 | 注入外部系统凭据 |

任何业务 Agent 不应直接绑定 Redis 或 PostgreSQL 具体客户端。业务代码只依赖这些抽象。

Fixture 规则：

```text
1. 每个 schema 至少有一个 valid fixture。
2. 关键失败场景必须有 invalid fixture。
3. e2e fixture 必须能串起事件、任务、诊断、通知和进度。
4. fixture 不允许包含真实密钥、真实 cookie 或生产日志原文。
```

## 13. 处理流程

```text
1. 定义通用字段 schema
2. 定义 ID、环境、时间、错误、artifact、metadata 契约
3. 定义敏感字段列表和脱敏规则
4. 定义基础设施抽象，不让业务代码直接绑定具体存储
5. 定义 schema 和 fixture 目录结构
6. 后续步骤引用这些契约，不重复定义
```

## 14. 与下一步衔接

`01-事件总线.md` 使用 `base.schema.json`、`id.schema.json`、`error.schema.json` 和 `artifact.schema.json` 定义 CloudEvents 扩展字段、事件归档、死信和投递记录。

`02-智能体运行时.md` 使用 `artifact.schema.json`、`runtime-policy.schema.json` 和 `error.schema.json` 定义任务输入输出、工具调用审计和失败降级。

Pipeline、Diagnosis、Collaboration 三类业务 Agent 必须复用 `correlation_id`、`trace_id`、`run_id`、`environment`、`artifact_ref` / `artifact_refs` 和 `error` 结构。

## 16. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 字段过早膨胀 | 只把跨两个以上模块依赖的字段提升为公共字段 |
| 后续新增字段 | 先进入 `metadata`，稳定后提升为 schema 字段 |
| 数据库尚未部署 | 使用 `EventArchiveRepository` 接口隔离，临时实现可写本地 artifact |
| Redis 尚未部署 | 使用 `IdempotencyStore` 接口隔离，临时实现可用内存或本地文件，但不能改变调用方接口 |
| Artifact 暂无对象存储 | 先使用本地相对路径，后续迁移对象存储 |
| 敏感字段漏脱敏 | 阻断写入模型输入和对外 artifact 摘要 |

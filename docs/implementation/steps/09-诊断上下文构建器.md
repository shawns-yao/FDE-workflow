# 09-诊断上下文构建器

## 1. 目标

把 ArgoCD、Kubernetes、Tekton、GitLab 的异常证据清洗、采样、压缩、结构化，生成 Diagnosis 三层漏斗可消费的标准上下文。Context Builder 是诊断质量的核心，不允许把全量日志直接交给 Claude API。

## 2. 不做什么

```text
不直接调用 Claude API
不直接生成最终根因
不保存无限量原始日志
不读取 K8s 写权限
```

## 3. 上游输入

```text
argocd.application.degraded
kubernetes.pod.failed
tekton.taskrun.completed(status=failed)
gitlab.pipeline.completed(status=failed)
build.fix.suggested（可选）
```

## 4. 输出结果

```text
diagnosis.context.built
diagnosis-context.json
证据 artifact 引用
```

## 5. 接口契约

### 5.1 输入证据

```json
{
  "application": "api-gateway",
  "environment": "dev",
  "argocd_status": {},
  "kubernetes_events": [],
  "pod_logs": [],
  "describe_snapshots": [],
  "build_logs": []
}
```

### 5.2 输出上下文

```json
{
  "context_id": "ctx-001",
  "application": "api-gateway",
  "environment": "dev",
  "symptoms": ["pod CrashLoopBackOff"],
  "timeline": [],
  "signals": {
    "pod_status": "CrashLoopBackOff",
    "restart_count": 8,
    "argocd_health": "Degraded"
  },
  "evidence_refs": []
}
```

## 6. 处理流程

```text
1. 收集 ArgoCD 状态
2. 收集 K8s Events
3. 收集 Pod logs 采样
4. 收集 describe 摘要
5. 合并 Tekton / GitLab 构建日志
6. 脱敏
7. 提取时间线和信号
8. 输出 diagnosis.context.built
```

## 7. 文件结构

```text
agents/
  diagnosis/
    context-builder/
      context-builder.md
      sampling-policy.md
schemas/
  diagnosis/
    diagnosis-context.schema.json
```

## 8. 与下一步衔接

`10-诊断规则引擎.md` 消费 `diagnosis.context.built`。规则未命中时，同一 context 进入知识库匹配和大模型根因分析。

## 9. 验证方式

```text
定向验证：输入 CrashLoopBackOff 样例日志，应输出 signals.pod_status。
定向验证：输出上下文不得包含 token、cookie、私钥。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| K8s 凭据未就绪 | 使用样例 evidence artifact |
| 日志过大 | 按 tail / since / error window 采样 |
| 证据缺失 | 输出 context_incomplete 标记 |

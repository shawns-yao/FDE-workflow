# 02-智能体运行时

## 1. 目标

定义 FDE 自有 Agent Runtime，作为 `@internal/agent-runtime` 这一内部库的能力边界。它不是外部 CLI 包装层，而是面向 Pipeline、Diagnosis、Collaboration 三类业务 Agent 的通用智能执行层。

这个 Runtime 负责把 Claude Code 中适合服务端和 CI 场景的通用能力迁移或重写到 FDE 内部，包括多轮模型调度、工具调用循环、上下文构建、文件与 Git 操作、受限 Shell 执行、结构化输出校验、权限控制、工具审计和 artifact 写入。

业务 Agent 负责事件消费、状态流转、领域规则和决策。Runtime 只负责执行"需要语义理解、文件操作或多步推理"的智能任务。

### 1.1 Agent、AI 与工具的关系

FDE 中的 Agent 不是普通定时脚本，也不是简单的大模型包装层。Agent 的成立条件是：

```text
有明确业务目标
能消费事件并维护状态
能根据规则和上下文选择下一步动作
能调用受控工具完成动作
在需要语义理解、生成、推理或代码修改时，能调用 AI 能力增强决策或执行
```

三个业务 Agent 都必须具备 AI 接入能力，但不要求每个业务步骤都调用 AI。确定性、安全敏感、可结构化表达的步骤优先使用纯代码；AI 只进入语义判断、复杂分析、内容生成、配置治理和修复建议等场景。

```text
Pipeline Agent：
  主链路用确定性代码执行，AI 用于 MR 评审、YAML 治理、构建失败修复。

Diagnosis Agent：
  数据采集和规则匹配用确定性代码，AI 用于日志摘要、根因推理、修复建议。

Collaboration Agent：
  通知路由、超时升级和状态追踪用确定性代码，AI 用于通知摘要、回复有效性判断、日报生成。
```

MCP 或 Runtime 工具不能在 Agent 之间直接流转。Agent 之间只流转标准化事件、artifact 引用和结构化结果；工具始终归 Runtime 管理，由 `permission_profile` 和 `allowed_tools` 控制。

```text
业务 Agent
  → 提交 Runtime task
    → Runtime 根据 permission_profile 暴露工具
      → 工具执行并生成 tool trace / artifact
        → Runtime 返回 TaskResult
          → 业务 Agent 发布 CloudEvent
            → 下一个 Agent 消费事件和 artifact
```

因此，工具流转不是“Pipeline 把 MCP 工具交给 Diagnosis”，而是“Pipeline 发布 artifact，Diagnosis 用自己的权限配置重新调用 Runtime 工具读取 artifact 或外部上下文”。

### 1.2 工具架构结论

`cc/` 的工具系统不是 MCP-only。它的实际结构是内置工具与 MCP 工具合并成一个模型可见工具池：文件、编辑、命令等本地能力是内置工具；MCP 负责外部系统和可扩展工具面。

FDE Runtime 必须采用同样的混合结构：

```text
CoreTool 统一协议
  ├─ BuiltinToolProvider：本地、安全敏感、低延迟工具
  └─ McpToolProvider：外部系统、跨平台集成、可扩展工具
        ↓
ToolRegistry / assembleToolPool
        ↓
permission_profile + allowed_tools 最终裁剪
        ↓
模型可见工具列表
```

不能把所有工具都改成 MCP。`read_file`、`edit_file`、`write_file`、`validate_schema`、`write_artifact`、`run_command` 这类工具必须贴近 Runtime 的沙箱、路径策略、权限审计和 artifact 管理。Tekton、ArgoCD、K8s、GitLab、飞书、知识库、浏览器、代码搜索等外部系统能力适合通过 MCP provider 或连接器 provider 扩展。

### 1.3 Agent Skill 与 MCP 的分层关系

FDE 借鉴 Agentic Skills 的组织方式，把“领域知识包”和“可执行工具”拆成两层：

```text
Agent Skill：
  定义某类 Agent 能力的触发条件、任务说明、上下文材料、Prompt 约束、输入输出 schema、推荐工具和验证方式。

MCP Tool：
  定义一个可执行外部系统动作，例如读取 Tekton 日志、查询 ArgoCD Application、更新飞书卡片。
```

Skill 不是工具调用权限本身。Skill 可以声明推荐工具，但 Runtime 必须继续按照 `permission_profile`、`allowed_tools`、环境策略和 MCP allowlist 裁剪模型可见工具。

```text
业务 Agent
  -> 选择 Skill 和业务任务类型
    -> Runtime 读取 Skill 上下文、schema 和推荐工具
      -> Runtime 根据权限装配 BuiltinToolProvider + McpToolProvider
        -> 模型只看到本次任务被允许的工具
```

第一批 Skill 不做独立市场或动态安装，只作为项目内可版本化能力包维护：

```text
skills/fde-pipeline/SKILL.md
skills/fde-diagnosis/SKILL.md
skills/fde-collaboration/SKILL.md
```

Skill 文件不得保存凭据，不得硬编码个人环境，不得直接封装 SDK 调用。所有外部系统动作必须进入 MCP provider、连接器或业务 Agent 的确定性控制器。

## 2. 不做什么

```text
不直接调用 Claude Code CLI 作为主路径
不保留 Claude Code 的 CLI / TUI / REPL 入口
不把业务规则写进 Runtime
不让 Runtime 直接消费 GitLab、Tekton、ArgoCD、K8s、Feishu 事件
不让 Runtime 管理业务状态机
不保留面向个人电脑的交互式登录态和自动更新逻辑
不把模型原始输出直接交给业务下游
不允许 prod 环境自动写入业务仓库
不把 Skill 当作可绕过权限的执行入口
不让 Skill 直接保存外部系统密钥或直接调用 SDK
```

## 3. 使用者与使用方式

### 3.1 Pipeline Agent

Pipeline Agent 是 Runtime 的主要代码执行使用者。

| 场景 | 是否调用 Runtime | 推荐任务类型 |
| --- | --- | --- |
| 镜像 Tag 提取 | 否 | 纯代码 |
| YAML Tag 更新 | 默认否 | 纯代码 + 结构化解析 |
| MR 语义评审 | 是 | `code_task` |
| YAML 配置治理 | 是 | `code_task` |
| 构建失败修复建议 | 是 | `repair_task` |
| 自动补丁生成 | 是 | `repair_task` |

Pipeline Agent 只把语义理解、补全、修复、评审这类任务交给 Runtime，其余步骤仍然走确定性代码。

### 3.2 Diagnosis Agent

Diagnosis Agent 主要使用分析能力。

| 场景 | 是否调用 Runtime | 推荐任务类型 |
| --- | --- | --- |
| 规则匹配 | 否 | 规则引擎 |
| 诊断上下文整理 | 可选 | `analysis_task` |
| 根因推理 | 是 | `analysis_task` |
| 日志摘要 | 是 | `analysis_task` |
| 修复建议生成 | 是 | `analysis_task` |

### 3.3 Collaboration Agent

Collaboration Agent 使用 Runtime 生成通知、摘要、升级文案和日报，但不使用代码编辑能力。

| 场景 | 是否调用 Runtime | 推荐任务类型 |
| --- | --- | --- |
| 飞书卡片摘要 | 是 | `analysis_task` |
| 回复有效性判断 | 是 | `analysis_task` |
| 升级文案生成 | 是 | `analysis_task` |
| 日报生成 | 是 | `analysis_task` |
| 飞书消息发送 | 否 | IM 连接器 |

### 4.1.2 第一批吸收范围

第一批只为 Pipeline Agent 的 YAML diff 审查和 YAML Governance 服务，不做通用桌面编码助手。

```text
必须进入第一批：
1. code_runtime 无头执行器骨架
2. CoreTool / registry / ToolProvider 工具协议
3. read_file / edit_file / write_file / list_files
4. run_command 命令白名单校验
5. structured-output 结构化结果工具
6. Anthropic Messages API 最小 client
7. tool trace / token usage / ErrorObject 映射

暂不进入第一批：
1. 终端 UI、REPL、交互权限弹窗
2. 个人配置驱动的第三方 MCP 动态加载和交互式鉴权
3. SkillTool / AgentTool / 子代理
4. WebSearch / WebFetch / 浏览器 / Computer Use
5. 本地记忆、插件市场、自动更新
6. Langfuse、Sentry、增长实验、遥测
7. 通用 BashTool
```

第一批验收条件：Pipeline Agent 在 YAML 更新后，把 `workspace_ref`、`yaml.diff`、构建上下文和治理 prompt 交给 `code_runtime`；Runtime 必须返回结构化审查结果，并能在高风险时阻断 Git 提交。

当前已落地的第一版 `code_runtime:code_task` 执行器具备以下能力：

```text
1. 读取 prompt_ref 和 artifact_refs
2. 通过 Anthropic Messages API 执行无头模型任务
3. 强制解析模型输出为 JSON
4. 使用 schema_ref 校验结构化结果
5. 写入 agent_task_result artifact
6. 返回标准 TaskResult，并把 schema 失败映射为 SCHEMA_VALIDATION_FAILED
7. 支持 Anthropic tool_use 最小循环
8. 支持 BuiltinToolProvider 和工具池装配
9. 支持 McpToolProvider 宿主适配接口和 `mcp__server__tool` 命名
10. 支持 read_file、list_files 两个受限工作区只读工具
11. 支持 run_command 受限命令工具，执行前按 permission_profile 的命令白名单校验
12. 写入 tool_trace artifact
```

当前尚未落地完整 Claude Code 工具调用循环。`edit_file`、`write_file`、`read_artifact`、`write_artifact`、`create_patch`、`validate_schema` 仍需要继续迁移为可被模型调用的受控工具；MCP provider 的宿主能力和 stdio transport 已落地，但 GitLab、Tekton、ArgoCD、K8s、飞书等真实外部系统 MCP server 仍需要按系统逐个实现。第一版 `code_runtime` 已具备只读上下文补充、受限命令查询和结构化风险判断能力；Pipeline YAML Governance 当前只开放 `read_file`、`list_files`，不开放 `run_command`。

### 4.2 不迁移的能力

```text
CLI 入口
终端交互 UI
用户本地配置管理
插件市场
个人配置驱动的第三方 MCP 自动加载
交互式记忆
自动更新
遥埋点
本地账号登录态
```

### 4.3 迁移方法

迁移不是整目录复制，而是按能力拆分。

```text
1. 找到 Claude Code 中对应能力的职责边界
2. 判断该能力是否依赖 CLI / TUI / 本地登录态
3. 判断许可证和维护方式是否允许迁移
4. 抽取接口，不保留 Claude Code 内部命名作为对外 API
5. 用 FDE 的 task、permission、artifact、schema 概念重新封装
6. 为迁移后的能力补 fixture、mock 和验证命令
```

如果某项能力不能稳定迁移，就参考其行为重新实现，不能让 Runtime 依赖不稳定内部结构。

### 4.4 迁移边界原则

```text
对外 API 只使用 FDE 命名：
  read_file / edit_file / write_file / list_files / run_command / validate_schema

内部实现可以吸收 `cc/`：
  但不能让业务 Agent 看到 FileReadTool、BashTool、SyntheticOutputTool 等 Claude Code 内部名称

权限由 FDE 决定：
  `permission_profile` 是强约束，`allowed_tools` 是本次任务的子集收窄

文件边界由 workspace_ref 决定：
  Runtime 不能访问 workspace_ref 外部路径，artifact 只能通过 artifact store 读写

Shell 不是通用入口：
  `run_command` 只允许白名单命令，不允许任意 Bash 脚本、管道、重定向、命令拼接

结构化输出必须可校验：
  code_runtime 的最终结果必须通过 `schema_ref` 校验后才能返回业务 Agent
```

## 5. Runtime 对外 API

### 5.1 标准接口

```typescript
interface AgentRuntime {
  runCodeTask(input: CodeTaskInput): Promise<TaskResult>;
  runAnalysisTask(input: AnalysisTaskInput): Promise<TaskResult>;
  runRepairTask(input: RepairTaskInput): Promise<TaskResult>;
}
```

### 5.2 通用输入

```typescript
interface BaseTaskInput {
  task_id: string;
  agent_type: "pipeline" | "diagnosis" | "collaboration";
  business_task_type:
    | "mr_review"
    | "yaml_governance"
    | "build_fix"
    | "log_triage"
    | "root_cause"
    | "notification"
    | "progress_tracking"
    | "daily_report";
  runtime_capability: "code_task" | "analysis_task" | "repair_task";
  runtime_type: "code_runtime" | "claude_api";
  workspace_ref?: string;
  context_refs: string[];
  artifact_refs?: string[];
  prompt_ref: string;
  schema_ref: string;
  permission_profile: string;
  runtime_policy: {
    environment: "dev" | "test" | "prod";
    timeout_ms: number;
    max_tool_calls: number;
    max_tokens: number;
    retry_count: number;
  };
  allowed_tools: string[];
  model: string;
  output_format: "json" | "text";
  correlation_id: string;
  trace_id: string;
  run_id: string;
}
```

### 5.3 任务结果

```typescript
interface TaskResult {
  task_id: string;
  status: "succeeded" | "failed" | "blocked" | "timed_out";
  output: string;
  structured_data?: unknown;
  artifact_refs: string[];
  patch_ref?: string;
  tool_trace_ref?: string;
  token_usage: {
    input_tokens: number;
    output_tokens: number;
  };
  permission_audit: {
    profile: string;
    blocked_tools: string[];
  };
  error?: ErrorObject;
}
```

`ErrorObject` 必须引用 `00-基础契约.md` 中的错误结构，字段包含 `code`、`message`、`retryable`、`severity`、`details`、`cause_ref`。Runtime 不允许重新定义缩减版错误结构。

### 5.3.1 错误码与状态映射

任务状态由 Runtime 根据错误码和执行结果自动推导，业务 Agent 不能直接指定状态。

| 错误码 | 任务状态 | 是否可重试 | 说明 |
| --- | --- | --- | --- |
| `SCHEMA_VALIDATION_FAILED` | `failed` | 否 | 输入或输出 schema 校验失败 |
| `UPSTREAM_UNAVAILABLE` | `timed_out` | 是 | 模型服务不可用或任务超时 |
| `AUTHENTICATION_FAILED` | `blocked` | 否 | 模型 API 认证失败 |
| `PERMISSION_DENIED` | `blocked` | 否 | 运行时权限不足 |
| `IDEMPOTENCY_CONFLICT` | `failed` | 否 | 幂等主键冲突 |
| `MODEL_NOT_CONFIGURED` | `blocked` | 否 | 执行器未配置 |
| `LLM_UNAVAILABLE` | `failed` | 是 | 大模型不可用 |
| `TOOL_PERMISSION_DENIED` | `blocked` | 否 | 工具调用权限被拒绝 |
| `ARTIFACT_WRITE_FAILED` | `failed` | 是 | Artifact 写入失败 |
| `EVENT_PUBLISH_FAILED` | `failed` | 是 | 事件发布失败 |
| `ARCHIVE_WRITE_FAILED` | `failed` | 是 | 归档写入失败 |
| `COMMAND_EXECUTION_FAILED` | `failed` | 是 | 命令执行失败（退出码非0） |
| `COMMAND_TIMEOUT` | `failed` | 否 | 命令执行超时或异常 |

状态推导规则：

```text
1. 工具权限拒绝 → blocked（不再提交）
2. 环境策略禁止写入 → blocked（prod 环境不执行写操作）
3. 执行器未配置 → blocked（等待运维配置）
4. 模型认证失败 → blocked（需修复凭据）
5. 超时 → timed_out（可重试）
6. 模型服务不可用 → timed_out（可重试）
7. 输出 schema 校验失败 → failed（需修复 prompt 或 schema）
8. 命令执行失败 → failed（可重试，除非退出码为126/127）
9. 命令执行超时 → failed（不可重试）
10. 其他运行时错误 → failed
11. 所有步骤成功 → succeeded
```

重试策略：

| 状态 | 重试条件 | 最大重试次数 |
| --- | --- | --- |
| `timed_out` | `retryable = true` 且未超过 `runtime_policy.retry_count` | 3 |
| `failed` | 仅 `retryable = true` 时允许 | 3 |
| `blocked` | 不允许重试 | 0 |

### 5.4 任务分类

```text
business_task_type：
业务维度标签，用于说明任务属于 MR 评审、YAML 治理、构建修复、日志分诊、根因分析、通知、进度追踪或日报生成。

runtime_capability：
能力维度标签，用于选择 Runtime 执行能力。

code_task：
面向仓库工作区，可使用文件、Git 和受限 Shell 工具。用于 MR 评审、YAML 治理、配置补全、代码变更建议。

analysis_task：
面向 artifact 和结构化上下文，默认不使用文件写入和 Shell。用于日志摘要、根因分析、回复判断、日报生成。

repair_task：
面向构建失败或代码修复，dev/test 可生成文件修改，prod 只能生成建议和 patch artifact。
```

业务任务和 Runtime 能力的推荐映射：

| business_task_type | runtime_capability | runtime_type |
| --- | --- | --- |
| `mr_review` | `code_task` | `code_runtime` |
| `yaml_governance` | `code_task` | `code_runtime` |
| `build_fix` | `repair_task` | `code_runtime` |
| `log_triage` | `analysis_task` | `claude_api` |
| `root_cause` | `analysis_task` | `claude_api` |
| `notification` | `analysis_task` | `claude_api` |
| `progress_tracking` | `analysis_task` | `claude_api` |
| `daily_report` | `analysis_task` | `claude_api` |

### 5.5 runtime_type 约定

`runtime_type` 是长期契约字段，不是临时过渡字段。它只表示执行后端，不表示业务任务。

```text
code_runtime：
FDE 自有代码任务运行时，迁移或重写 Claude Code 的通用编码 Agent 能力，不通过 CLI 子进程作为主路径。

claude_api：
直接调用 Claude API，用于日志摘要、根因推理、通知生成、回复判断和日报生成。
```

执行器选择必须同时使用 `runtime_type` 和 `runtime_capability`。例如 `code_runtime:code_task` 选择代码任务无头执行器，`claude_api:analysis_task` 选择直接 Claude API 分析执行器。Runtime 入口方法还必须校验 `input.runtime_capability` 与当前入口一致；不一致时返回 `CONFIGURATION_INVALID`，不能静默按入口方法覆盖输入字段。

## 6. 工具封装

Runtime 内部统一使用自己的工具名，不把 Claude Code 的工具名直接暴露给业务 Agent。

### 6.1 Skill 装配

Runtime 接收任务时可以加载一个或多个项目内 Skill。Skill 提供上下文和约束，不能替代任务 schema。

```json
{
  "skill_name": "fde-pipeline",
  "business_task_type": "yaml_governance",
  "runtime_capability": "code_task",
  "prompt_ref": "skills/fde-pipeline/yaml-governance.md",
  "output_schema_ref": "schemas/pipeline/yaml-governance-result.schema.json",
  "recommended_tools": ["read_file", "list_files"]
}
```

装配规则：

```text
1. Skill 的 business_task_type 必须与 task.business_task_type 匹配。
2. Skill 的 runtime_capability 必须与 task.runtime_capability 匹配。
3. Skill 的 recommended_tools 只能作为 allowed_tools 的候选来源，不能扩大权限。
4. 若 task.allowed_tools 为空，Runtime 可以按 profile 与 Skill 推荐工具取交集。
5. 若 task.allowed_tools 非空，必须继续校验它是 permission_profile 的子集。
6. Skill 引用的 prompt、schema 和 fixture 必须使用仓库内稳定路径。
```

Runtime 工具来自两个 provider：

```text
builtin provider:
- Runtime 内部实现
- 负责工作区文件、artifact、schema、命令白名单、tool trace
- 必须受 workspace_ref、permission_profile、runtime_policy 控制

mcp provider:
- Runtime 通过 MCP client 或连接器适配器发现
- 负责 Tekton、ArgoCD、K8s、GitLab、飞书、知识库、浏览器等外部系统
- 工具名统一为 mcp__<server>__<tool>
- 必须先经过 permission_profile / allowed_tools 裁剪，再暴露给模型
```

业务 Agent 不直接声明 MCP server。业务 Agent 只声明 `allowed_tools`，Runtime 根据 profile、环境和 provider 配置装配工具池。内置工具与 MCP 工具同名时，内置工具优先，避免外部工具覆盖本地安全边界。

MCP provider 必须具备以下生命周期能力：

```text
start:
- 执行鉴权
- 建立 MCP client 或连接器连接
- 鉴权失败映射为 AUTHENTICATION_FAILED

refreshTools:
- 从外部 server 重新发现工具
- 刷新本地工具定义缓存
- 发现失败映射为 UPSTREAM_UNAVAILABLE 或 AUTHENTICATION_FAILED

listTools:
- 使用缓存或触发刷新
- 只返回 allowed_tools 当前声明的工具

call:
- 调用外部工具
- 调用失败必须返回 ErrorObject，不允许把原始异常直接抛给业务 Agent

stop:
- 关闭连接
- 清空工具缓存
```

当前代码已落地 MCP provider 宿主适配层、生命周期接口、stdio MCP transport 和 `FDE_RUNTIME_MCP_SERVERS` 配置加载。stdio transport 支持 MCP initialize、`tools/list`、`tools/call`、`Content-Length` framing、请求超时、必需环境变量鉴权检查和断开连接。HTTP / SSE transport、OAuth / token 刷新、工具发现后台刷新和真实外部系统 MCP server 仍属于后续外部系统接入工作。

stdio MCP server 通过 `FDE_RUNTIME_MCP_SERVERS` 环境变量配置，值为 JSON 数组。Runtime 在创建 `code_runtime` 执行器时会读取该配置并装配 provider；如果当前任务的 `allowed_tools` 不包含对应 `mcp__<server>__<tool>`，provider 不会启动外部 server。

```json
[
  {
    "transport": "stdio",
    "server_name": "argocd",
    "command": "node",
    "args": ["dist/mcp/argocd-server.js"],
    "required_env": ["ARGOCD_TOKEN"],
    "request_timeout_ms": 30000,
    "refresh_interval_ms": 60000
  }
]
```

### 6.3 工具权限

工具权限由 `permission_profile` 决定，业务 Agent 只能选择预定义配置，不能在任务中临时追加任意工具。`allowed_tools` 只能作为 `permission_profile` 允许工具集合的子集，用于进一步收窄本次任务权限；如果 `allowed_tools` 包含 profile 未授权工具，Runtime 必须拒绝任务。

当前代码实现的模型可见内置工具为 `read_file`、`list_files`、`run_command`。权限 profile 中列出的其他内置工具是目标契约，必须等对应 provider 工具实现后才能在业务 Agent 的 `allowed_tools` 中启用。MCP 工具名通过 schema pattern 支持 `mcp__<server>__<tool>`，并由 `mcp_server_allowlist` 或 `mcp_tool_allowlist` 做二次授权。

```text
ci-readonly:
- read_file
- list_files
- run_command 仅允许 rg、git diff、git show、git status
- read_artifact
- write_artifact
- validate_schema
- MCP server：gitlab、tekton、argocd、kubernetes

ci-yaml-edit:
- read_file
- edit_file
- write_file
- list_files
- run_command 仅允许 rg、git diff、git show、git status、yq、kustomize build
- create_patch
- read_artifact
- write_artifact
- validate_schema
- MCP server：gitlab、tekton、argocd、kubernetes

diagnosis-readonly:
- read_artifact
- write_artifact
- summarize_log
- reason_about_failure
- validate_schema
- redact_sensitive_fields
- MCP server：gitlab、tekton、argocd、kubernetes

collaboration-notify:
- read_artifact
- write_artifact
- classify_reply
- draft_notification
- validate_schema
- redact_sensitive_fields
- MCP server：feishu、gitlab
```

命令级白名单必须在 `run_command` 工具层校验。Runtime 不允许把原始命令直接交给 Shell 后再依赖 Shell 返回结果判断权限；命令名称、参数、工作目录和环境变量必须先通过 `check_permission`。

### 6.4 生产环境策略

```text
dev:
- 允许在受限工作区内自动写文件
- 允许生成 patch

test:
- 允许在受限工作区内自动写文件
- 必须输出 diff artifact

prod:
- 禁止 write_file 和 edit_file
- 只能输出建议、patch artifact 和人工确认步骤
- 禁止执行部署变更命令
```

## 7. 处理流程

```text
1. 接收业务 Agent 提交的 task
2. 校验 task schema
3. 根据 runtime_capability、runtime_type 和 runtime_policy 选择执行器
4. 装载 prompt、context 和 artifact
5. 绑定 workspace_ref，并限制文件访问范围
6. 检查 allowed_tools 和 permission_profile
7. 执行多轮模型调度和工具调用
8. 记录 tool trace
9. 对输出执行 JSON schema 校验
10. 写入 result、patch、trace、usage 等 artifact
11. 返回 TaskResult
```

## 11. 风险与降级

| 风险 | 降级 |
| --- | --- |
| Claude Code 内部模块不稳定 | 参考行为重新实现，不依赖内部 API |
| 许可证不允许迁移代码 | 只迁移设计思想和接口，不复制实现 |
| 模型不可用 | 返回 `LLM_UNAVAILABLE`，业务 Agent 进入规则或人工处理 |
| 输出 JSON 不合法 | 保存原始输出 artifact，返回 schema validation error |
| 工具越权请求 | 拒绝工具调用，记录 permission audit |
| prod 环境出现写入请求 | 阻断写入，输出 patch artifact 和人工确认步骤 |
| 上下文过大 | 启用 context_compactor，保留 evidence_refs |

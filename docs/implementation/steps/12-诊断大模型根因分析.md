# 12-诊断大模型根因分析

## 1. 目标

实现 Diagnosis 三层漏斗的第三层：Claude API 根因分析。仅当前两层未命中或置信度不足时触发，用结构化上下文和相似案例做复杂推理。

## 2. 不做什么

```text
不直接读取无限量原始日志
不修改代码或配置
不直接通知人员
不输出无 evidence 的结论
```

## 3. 上游输入

```text
diagnosis.knowledge.missed
diagnosis.context.built
knowledge_refs（可选）
```

## 4. 输出结果

```text
diagnosis.completed
knowledge.case.candidate
llm-root-cause-report.json
```

## 5. 接口契约

### 5.1 LLM 输入

```json
{
  "context": {},
  "similar_cases": [],
  "required_output_schema": "schemas/diagnosis/diagnosis-result.schema.json"
}
```

### 5.2 LLM 输出

```json
{
  "diagnosis_id": "diag-002",
  "source": "claude_api",
  "funnel_layer": "llm",
  "category": "environment",
  "severity": "high",
  "summary": "服务启动依赖的配置中心不可达",
  "root_cause": "Pod 内部连接配置中心超时",
  "impact": "新版本不可用，旧版本仍服务",
  "recommendation": "检查网络策略和配置中心地址",
  "confidence": 0.86,
  "evidence_refs": [],
  "case_candidate": {}
}
```

## 6. 处理流程

```text
1. 读取结构化 context
2. 拼接 prompt，不包含敏感原文
3. Agent Runtime 使用 claude_api
4. 校验 diagnosis-result schema
5. 输出 diagnosis.completed
6. 生成 knowledge.case.candidate，等待审核
```

`source` 与 `funnel_layer` 必须同时输出。Claude API 生成的诊断结果固定为 `source=claude_api`、`funnel_layer=llm`；当 Claude API 不可用并使用规则兜底时，结果必须改为 `source=fallback_rule`，不能仍标记为 `claude_api`。

## 7. 文件结构

```text
agents/
  diagnosis/
    root-cause/
      llm-root-cause.md
prompts/
  diagnosis/
    root-cause.md
schemas/
  diagnosis/
    diagnosis-result.schema.json
    knowledge-case-candidate.schema.json
```

## 8. 与下一步衔接

诊断完成后进入 `13-协同通知路由.md`。`case_candidate` 后续进入知识库或记忆审核链路，不在本步骤自动批准。

## 9. 验证方式

```text
定向验证：用 mock Claude API 输出校验 schema。
定向验证：输出必须包含 evidence_refs。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| Claude API 不可用 | 输出 fallback diagnosis，标记 llm_unavailable |
| 输出不符合 schema | 保存原始输出，返回 validation error |
| 推理不确定 | confidence 低时通知中标记“需要人工确认” |

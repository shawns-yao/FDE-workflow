# 11-诊断知识库匹配

## 1. 目标

实现 Diagnosis 三层漏斗的第二层：知识库匹配。规则未命中时，用结构化上下文匹配历史案例，复用已有根因和处理方案。

M1 本层只做 stub 和 schema，不引入数据库、向量检索或长期知识库服务。相似度分数只能来自 fixture 中的显式字段或简单关键词规则，用于打通链路；真正的向量匹配、案例审核和长期沉淀推迟到 M2。

## 2. 不做什么

```text
不直接调用 Claude API
不自动写入最终知识库
不在无审核情况下沉淀高风险经验
不返回无证据的历史方案
M1 不接入向量数据库
M1 不选择相似度引擎
```

## 3. 上游输入

```text
diagnosis.rule.missed
diagnosis.context.built
history case candidates
```

## 4. 输出结果

```text
diagnosis.completed（高相似命中）
diagnosis.knowledge.missed（未命中）
knowledge-match-report.json
```

## 5. 接口契约

### 5.1 知识案例

```json
{
  "case_id": "case-001",
  "symptom": "Pod 启动后反复重启",
  "root_cause": "启动命令缺少环境变量",
  "solution": "补充 CONFIG_PATH 环境变量",
  "evidence_refs": [],
  "tags": ["CrashLoopBackOff", "env"],
  "owner_hint": "backend"
}
```

### 5.2 匹配输出

```json
{
  "matched": true,
  "stub_mode": true,
  "similarity": 0.91,
  "case_id": "case-001",
  "diagnosis_result": {
    "source": "knowledge",
    "funnel_layer": "knowledge"
  }
}
```

## 6. 处理流程

```text
1. 读取 diagnosis context
2. 提取症状、信号、错误关键词
3. M1 从 fixture 或本地 JSON 中读取历史案例
4. 使用显式 similarity 字段或关键词规则生成 stub 匹配结果
5. 相似度超过阈值则复用方案
6. 未命中则进入 LLM 根因分析
```

M1 阶段本步骤的实现边界是“让诊断链路有知识库层的契约和跳转点”，不是完成知识库产品。若没有 fixture，必须输出 `diagnosis.knowledge.missed`，不能阻塞后续 LLM 根因分析。

## 8. 与下一步衔接

高相似命中时进入 `13-协同通知路由.md`。未命中时进入 `12-诊断大模型根因分析.md`。LLM 生成的新案例作为候选回流本步骤。

## 9. 验证方式

```text
定向验证：fixture 中显式 similarity 超过 0.85 时输出 matched=true。
定向验证：未命中时输出 diagnosis.knowledge.missed。
定向验证：无 fixture 或知识库为空时直接进入 LLM 根因分析。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 知识库为空 | 直接进入 LLM 根因分析 |
| 相似度误判 | 设置高阈值，低置信度进入 LLM |
| 案例过旧 | 使用 case metadata 标记有效期 |

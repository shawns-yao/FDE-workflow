# 03-飞书连接器

## 1. 目标

定义 IM 连接器，作为 FDE Workstation 的统一消息出口和交互入口。它是纯管道，只负责飞书消息发送、卡片更新、回复接收、用户/群组映射、回调标准化和事件回流，不负责任何业务判断。

业务 Agent 只能通过该连接器触达飞书，不直接散落调用飞书 SDK 或 HTTP API。

## 2. 不做什么

```text
不判断通知谁
不生成通知内容
不处理业务按钮逻辑
不保存业务状态机
不把飞书消息当业务数据库
不在消息中输出原始敏感日志
不绕过事件总线直接处理回调业务
不绕过审批直接执行生产变更
```

职责归属：

```text
通知对象由 Collaboration Agent 的 notification-router 决定。
通知内容由 Collaboration Agent 根据诊断结果、日报数据和模板策略生成；需要语义摘要时，通过 Agent Runtime 的 analysis_task 调用 Claude API。
飞书连接器只接收已确定的 target、mentions、card_type、title、summary、actions 和 data，并负责投递、更新卡片和回调标准化。
```

## 3. 使用者与使用方式

### 3.1 Collaboration Agent

Collaboration Agent 是 IM 连接器的主要使用者，负责通知编排、升级判断、回复判断和日报落地。

### 3.2 Diagnosis Agent

Diagnosis Agent 生成诊断结果后，通常通过 Collaboration Agent 发起通知。若 Diagnosis Agent 直接调用 IM 连接器，必须传入已由上游策略确定的目标、摘要和证据引用，不能让连接器判断通知对象或生成通知内容。

### 3.3 Pipeline Agent

Pipeline Agent 在构建失败、配置治理和部署异常时，可以通过 IM 连接器投递状态卡片，但通知对象、升级策略和通知内容必须来自 Collaboration Agent 或预先定义的通知策略。

## 4. 技术选型

| 类别 | 选型 | 说明 |
| --- | --- | --- |
| 开发语言 | TypeScript | 与 Runtime 和事件总线统一 |
| 飞书 SDK | `@larksuiteoapi/node-sdk` | 用于 OpenAPI 发送和长连接事件接入，业务 Agent 不直接依赖 |
| 部署形态 | 独立微服务 | 通过 REST API 提供给业务 Agent 调用 |
| 事件入口 | SDK 长连接为主，HTTP callback 兼容 | 个人飞书和单服务器部署优先使用长连接，仅需出站 443 |

SDK 和 HTTP 规则：

```text
1. 对外契约以本步骤为准，不以 SDK 类型为准。
2. SDK 只能存在于 connectors/feishu 内部。
3. 业务 Agent 不得直接依赖 SDK 类型。
4. 连接器可以选择 HTTP 封装或官方 SDK，但对外输出必须稳定。
5. 飞书事件入口由 FEISHU_EVENT_MODE 控制，默认使用 SDK WebSocket 长连接。
```

## 5. 核心职责

```text
消息发送
卡片更新
回复接收
回调签名校验
用户 / 群组映射
卡片动作标准化
事件回流总线
发送结果归档
```

### 5.1 接入形态

飞书连接器支持两种接入模式：

| 形态 | 适用功能 | 是否支持 @ 人 | 是否支持按钮回调 | 需要内容 |
| --- | --- | --- | --- | --- |
| 自定义机器人 webhook | 环境检测报告、日报、简单告警 | 有限制，按能力做 best effort | 不支持 | webhook 地址、签名密钥、默认群 |
| 飞书应用机器人 OpenAPI | 故障通知、@责任人、按钮认领、进度追踪、升级闭环 | 支持，当前发送目标使用 open_id | 支持，默认走 SDK 长连接 | app_id、app_secret、机器人权限、事件订阅 |

自定义机器人 webhook 只适合验证单向通知和卡片 JSON。只要需要 `acknowledge`、`claim`、`mark_fixed`、回复识别或升级闭环，就必须使用飞书应用机器人 OpenAPI。

OpenAPI 模式下，事件入口有三种：

```text
websocket：默认模式，官方 SDK 主动向飞书建立 WebSocket 长连接；服务器只需要出站公网 443，不需要公网入站端口、域名或证书。
http_callback：兼容模式，飞书通过公网 HTTPS 回调连接器；需要公网域名、证书、回调 token 和签名密钥。
disabled：不接收飞书事件，只保留主动发送能力。
```

### 6.1 发送卡片输入

```typescript
interface SendCardInput {
  mode: "webhook_bot" | "openapi_bot";
  target_type: "chat" | "group" | "user";
  target_id: string;
  mentions?: Array<{
    type: "user" | "group";
    id: string;
    reason: string;
  }>;
  card_type:
    | "environment_scan_report"
    | "diagnosis_notification"
    | "pipeline_build_notification"
    | "mr_review_report"
    | "escalation_notice"
    | "daily_report"
    | "custom";
  title: string;
  summary: string;
  severity: "low" | "medium" | "high" | "critical";
  actions?: Array<{
    type: "open_url" | "acknowledge" | "claim" | "mark_fixed" | "escalate" | "open_artifact";
    label: string;
    url?: string;
    value?: string;
  }>;
  data: Record<string, unknown>;
  correlation_id: string;
  trace_id: string;
  run_id: string;
}
```

`schemas/connectors/feishu/card.schema.json` 校验完整 `SendCardInput`，包含 `mode`、`target_type`、`target_id`、`mentions`、`card_type`、`actions` 和链路字段；它不是只校验飞书卡片渲染体。

OpenAPI 发送目标映射必须与当前代码一致：

```text
target_type = user -> receive_id_type = open_id，target_id 必须是用户 open_id
target_type = chat -> receive_id_type = chat_id，target_id 必须是会话 chat_id
target_type = group -> receive_id_type = chat_id，target_id 必须是群 chat_id
```

当前接口不支持把 `union_id` 作为发送目标。若后续需要支持 `union_id`，必须新增显式的 `target_id_type` 字段或独立解析层，不能让 `target_id` 同时承载多种 ID 语义。

OpenAPI 卡片 `mentions` 的当前实现规则：

```text
mentions.type = user 时，id 必须是当前应用下的用户 open_id。
连接器将 user mention 渲染为飞书 lark_md：<at id=ou_xxx></at>。
mentions.type = group 当前不自动渲染为 @ 群，不用于 @ 全体。
@ 全体默认不开放，避免误通知。
```

`open_id` 获取方式：

```text
1. 由 Collaboration Agent 的责任人映射直接提供 feishu_open_id。
2. 用户主动给机器人发消息时，可从飞书消息事件的 sender_id.open_id 获取。
3. 应用具备群成员读取权限时，可通过 chat_id 读取群成员 open_id。
4. 应用具备通讯录权限时，可通过邮箱、手机号或用户 ID 转换得到 open_id。
```

连接器不能仅凭 `chat_id` 推断“应该 @ 谁”。读取群成员也只解决“这个群里有哪些 open_id”，不负责决定通知对象；通知对象仍由 Collaboration Agent 或显式配置决定。

## 7. 事件回流闭环

用户在飞书卡片上点击按钮、回复消息或认领任务时，连接器负责把飞书事件转成标准事件，再发回事件总线。默认链路使用 SDK 长连接：

```text
IM 连接器使用 SDK 建立 WebSocket 长连接
  -> 飞书通过长连接推送消息事件或卡片点击事件
  -> 解析 action / reply / operator / message_id
  -> 生成标准 CloudEvent
  -> 发布到事件总线
  -> 对应业务 Agent 消费
  -> 处理完成后再调用 IM 连接器更新卡片状态
```

兼容 HTTP callback 模式时，第一步改为：

```text
飞书推送 HTTP 回调事件
  -> IM 连接器校验 token / 签名
  -> 后续标准化和事件发布流程保持一致
```

标准事件类型：

```text
feishu.card.action_clicked
feishu.message.replied
```

回调事件必须包含：

```text
message_id
action
operator
raw_callback_excerpt
correlation_id
trace_id
run_id
```

自然语言回复只保存脱敏摘要和 message_id，不保存包含敏感日志的全文。

## 8. 可更新状态卡片

运维场景的核心体验是同一条卡片持续更新，不刷屏。

典型状态流转：

```text
诊断中 -> 诊断完成 -> 处理中 -> 已闭环
```

连接器必须支持按 `message_id` 更新同一条卡片的标题、摘要、动作和状态标签。

凭据只能来自本地 `.env`、进程环境变量或 CI / Secret 管理系统，不写入仓库。`.env` 用于本地开发，必须被 `.gitignore` 忽略；`.env.example` 可以提交，但只能包含变量名和空占位值。项目服务启动入口先加载 `.env`，再由业务代码从 `process.env` 读取运行时配置。发送结果 artifact 只能保存脱敏后的 target 摘要，不能保存 secret、token 或原始回复全文。

### 9.1 项目服务启动入口

飞书回调闭环必须通过项目服务入口承载，不能用 smoke 命令代替。

启动命令：

```text
npm run start
```

项目服务启动后暴露以下基础端点：

```text
GET /health
GET /ready
POST /webhook/gitlab
POST /webhook/tekton
POST /webhook/argocd
POST /webhook/kubernetes
POST /webhook/feishu/callback（仅 FEISHU_EVENT_MODE=http_callback 时作为飞书事件入口）
```

默认 `FEISHU_EVENT_MODE=websocket` 时，项目服务启动后会在同一进程内启动飞书 SDK 长连接。该连接由服务端主动向飞书发起，飞书通过这条通道推送消息和卡片点击事件。`/webhook/feishu/callback` 只保留给 HTTP callback 兼容模式。

如配置 `FDE_FEISHU_STARTUP_MESSAGE`，项目服务启动成功后会通过服务内的飞书连接器发送一条测试卡片。目标群优先级如下：

```text
FEISHU_STARTUP_MESSAGE_CHAT_ID
FEISHU_TEST_CHAT_ID
FEISHU_DEFAULT_CHAT_ID
```

该能力用于验证“项目服务进程内发送飞书消息”，不同于 `feishu:send-smoke` 辅助命令。

启动测试卡片支持以下可选配置：

```text
FEISHU_STARTUP_MENTION_OPEN_IDS：逗号分隔的用户 open_id，显式 @ 指定人。
FEISHU_STARTUP_MENTION_FROM_CHAT_MEMBERS=true：从目标群读取群成员 open_id 后 @ 前 N 个成员。
FEISHU_STARTUP_MENTION_LIMIT：从群成员读取时的 @ 人数量，默认 1，最大 10。
FEISHU_STARTUP_ACTION_URL：添加一个 open_url 按钮。
FEISHU_STARTUP_ACTION_LABEL：open_url 按钮文案，默认“查看详情”。
FEISHU_STARTUP_ENABLE_CALLBACK_ACTIONS=true：OpenAPI 模式下额外添加 acknowledge 按钮。
FEISHU_STARTUP_ACK_LABEL：acknowledge 按钮文案，默认“确认收到”。
```

`FEISHU_STARTUP_MENTION_OPEN_IDS` 优先级高于 `FEISHU_STARTUP_MENTION_FROM_CHAT_MEMBERS`。如果启用群成员读取但飞书应用缺少权限，启动消息发送结果必须返回结构化 `failed`，不能静默发送一张缺少 @ 的卡片。

启动测试消息必须异步发送。服务启动成功的判定以 HTTP 服务成功监听和 `/health` 可用为准，不能等待飞书发送成功后才认为服务启动成功。飞书是外部系统，网络异常、权限变化、机器人退群、限流或配置错误都不应阻断 FDE 服务启动。

异步发送不能静默失败，必须满足以下可观测要求：

```text
1. 服务启动日志先输出 started。
2. 启动消息发送完成后输出 feishu_startup_message_result。
3. 成功时输出 status=sent、message_id、target_id、sent_at。
4. 失败时输出 status=failed、target_id 和标准 ErrorObject。
5. 错误详情必须脱敏，不能输出 App Secret、tenant_access_token 或 Authorization。
```

启动测试消息只用于服务内飞书发送链路验证，不进入可靠投递语义。正式业务通知必须走事件驱动链路，由业务 Agent 生成通知请求，Collaboration Agent 消费后调用 IM 连接器，并记录 `notification.sent` / `notification.failed`、重试和死信状态。

OpenAPI 应用机器人向群发送消息前，应用机器人必须已经加入目标群。若飞书返回：

```text
code = 230002
msg = Bot/User can NOT be out of the chat.
```

说明当前 `chat_id` 对应的群里没有该应用机器人，需先在飞书群中添加应用机器人，再重新启动服务验证。

### 9.2 OpenAPI smoke 联调

smoke 命令只用于验证应用机器人发卡片主路径，不是项目服务入口，也不能代表飞书闭环完成。它不验证按钮回调、责任人映射或消息回复回流。

必需环境变量：

```text
FEISHU_APP_ID
FEISHU_APP_SECRET
FEISHU_TEST_CHAT_ID
```

本地开发使用仓库根目录下的 `.env` 注入以上变量；`.env.example` 只作为模板，不填写真实密钥。CLI 启动时加载 `.env` 后再从 `process.env` 读取配置。

运行命令：

```text
npm run feishu:send-smoke
```

约束：

```text
App Secret 只能写入本地 `.env` 或 Secret 管理系统，不能写入 `.env.example`、文档、fixture、代码或提交记录。
smoke 命令只打印 SendCardResult，不打印 App Secret、tenant_access_token 或原始请求体。
FEISHU_TEST_CHAT_ID 只用于测试群发卡片验证；正式通知目标仍由业务 Agent 或协同路由生成。
```

## 10. 处理流程

```text
1. 项目服务启动时加载 .env，并初始化事件总线、飞书连接器和 HTTP 服务
2. 发送链路根据 FEISHU_MODE 选择 webhook_bot 或 openapi_bot
3. 发送链路校验目标人、群和 mentions
4. 发送链路生成飞书卡片结构并调用 webhook 或 OpenAPI
5. 发送链路保存发送结果 artifact
6. 事件入口根据 FEISHU_EVENT_MODE 选择 websocket、http_callback 或 disabled
7. websocket 模式由 SDK 长连接接收飞书事件；http_callback 模式由 POST /webhook/feishu/callback 接收飞书事件
8. 事件链路解析 action / reply / operator / message_id
9. 飞书事件映射为 feishu.* CloudEvent
10. CloudEvent 发布到事件总线，由业务 Agent 消费
```

`webhook_bot` 模式不支持 `acknowledge`、`claim`、`mark_fixed`、`escalate` 这类交互动作。连接器收到这些 action 时必须拒绝请求并返回结构化错误，或按显式配置降级为只保留 `open_url`；不得静默忽略。

## 14. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 飞书凭据未就绪 | 输出 Markdown 通知 artifact |
| 长连接不可用 | 记录 ErrorObject，保留主动发送；必要时切到 http_callback 兼容模式 |

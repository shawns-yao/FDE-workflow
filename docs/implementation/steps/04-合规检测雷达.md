# 04-合规检测雷达

## 1. 目标

定义合规检测雷达，作为整套 Agent 平台的基础设施健康守门员。它周期性、主动式探测内部依赖平台的 API 可用性、权限有效性和基础配置合规性，输出标准化环境健康报告，并把异常事件发布到事件总线。

它的核心价值是提前发现 GitLab、Tekton、ArgoCD、K8s 等依赖平台的故障和权限漂移，而不是等 Pipeline、Diagnosis、Collaboration 执行失败后再被动排查。

本步骤属于平台级共享基础设施，不是业务 Agent，也不是 MR 代码合规扫描。

## 2. 不做什么

```text
不做业务代码安全扫描
不做 MR 语义评审
不做实时业务监控和指标告警
不做故障根因分析
不自己发飞书通知
不修改 GitLab、Tekton、ArgoCD 或 K8s 资源
不把 API 不可用伪装成可降级成功
```

通知、升级和责任人判断交给 Collaboration Agent。故障根因分析交给 Diagnosis Agent。雷达只输出健康报告和标准事件。

## 3. 技术选型

| 类别 | 选型 | 说明 |
| --- | --- | --- |
| 开发语言 | TypeScript / Node.js | 与事件总线、Agent Runtime、IM 连接器统一 |
| 服务框架 | NestJS 或 Fastify | 推荐 NestJS，便于模块化探测器和调度器 |
| 定时调度 | NestJS Schedule 或 node-cron | 支持周期巡检 |
| GitLab | `@gitbeaker/rest` 或封装 REST API | 检测 API、权限、MR、Pipeline、Webhook 能力 |
| Tekton / K8s | `@kubernetes/client-node` | 检测 CRD、PipelineRun、TaskRun、命名空间和 RBAC |
| ArgoCD | 封装 ArgoCD REST API | 检测 token、Application、健康状态、同步能力 |
| 结果存储 | PostgreSQL | 保存历史检测记录，供诊断、日报和运营分析使用 |
| 事件输出 | 事件总线 | 异常和完成事件统一发布为 CloudEvent |
| 报告输出 | JSON + Markdown artifact | JSON 供程序消费，Markdown 供人工阅读 |

## 6. 整体架构

```text
触发层
  -> 定时调度 / 手动触发 / Pipeline 前置预检
引擎层
  -> 调度控制 / 策略加载 / 结果聚合 / 事件发布
探测层
  -> GitLab Probe / Tekton Probe / ArgoCD Probe / Kubernetes Probe
输出层
  -> JSON 报告 / Markdown 报告 / PostgreSQL 历史记录 / CloudEvent
```

模块职责：

| 模块 | 职责 |
| --- | --- |
| `radar-scheduler` | 执行周期性扫描 |
| `radar-controller` | 提供手动触发和前置预检入口 |
| `radar-engine` | 调度探测器、聚合结果、生成状态 |
| `probe-registry` | 管理各系统探测器 |
| `report-writer` | 输出 JSON 和 Markdown artifact |
| `radar-history-repository` | 保存历史检测记录 |

## 7. 触发方式

| 触发方式 | 说明 | 使用者 |
| --- | --- | --- |
| 定时巡检 | 默认每小时一次，全量检测所有依赖 | 平台运维 |
| 手动触发 | 调试、凭据变更、平台升级后主动执行 | 运维、开发 |
| 前置预检 | Pipeline 构建执行前检查关键依赖 | Pipeline Agent |
| 事件触发 | 凭据轮换、配置变更后自动执行目标系统检测 | 事件总线 |

前置预检必须支持快速模式，只检测当前流程依赖的目标系统和关键权限，避免构建前等待过久。

## 8. 检测范围与检测项

检测分三层：连通性检测、权限与功能检测、配置合规检测。文档定义完整目标范围，具体实现可按探测器逐步落地，但 schema 和报告结构不能缩水。

实现优先级说明：

```text
1. 首批实现必须覆盖每个依赖系统的连通性检测和关键权限检测，保证 Pipeline 前置预检可以判断是否阻断。
2. 配置合规检测可按风险顺序逐项接入，但报告结构必须保留 configuration 层。
3. 尚未接入的检测项不能标记为 healthy，必须在报告 metadata.pending_checks 中列明。
4. 本文不按里程碑拆分检测表；下表始终表示完整目标契约。
```

| 依赖系统 | 一级：连通性检测 | 二级：权限与功能检测 | 三级：配置合规检测 |
| --- | --- | --- | --- |
| GitLab | 域名和端口可达；基础 API 返回正常 | Access Token 有效且未过期；仓库读写权限；MR 评论权限；Webhook 配置读取权限；Pipeline 读取权限 | Token 非永不过期；机器人账号最小权限；关键仓库开启分支保护 |
| Tekton | K8s API Server 可达；Tekton CRD 已安装 | ServiceAccount 具备 PipelineRun / TaskRun 读写权限；可读取目标 namespace；可执行测试 TaskRun | Task 配置资源限制；工作目录隔离；禁止高危特权容器 |
| ArgoCD | ArgoCD Server API 可达；核心组件健康 | API Token 有效；可查询 Application 列表与状态；可触发同步能力探测 | 应用开启健康检查；自动同步策略符合规范；无越权 Application 配置 |
| K8s | API Server 连通；核心组件健康 | ServiceAccount 权限有效；节点状态正常；核心命名空间 Pod 正常；Event 读取权限 | 命名空间资源配额；禁止默认 ServiceAccount 挂载令牌；RBAC 权限最小化 |

## 9. 结果分级

所有检测项统一使用三级状态。

```text
healthy：正常，检测通过
warning：警告，不阻断核心流程，但存在风险
critical：严重，核心能力不可用，业务流程可能失败
```

总体状态计算：

```text
任一 critical -> overall_status = critical
无 critical 且存在 warning -> overall_status = warning
全部 healthy -> overall_status = healthy
```

Pipeline 前置预检中，`critical` 必须阻断真实构建或部署流程。`warning` 是否阻断由 `radar_policy_config` 决定。

## 11. 事件联动

扫描完成后必须发布标准 CloudEvent。

```text
全部 healthy：
- compliance.environment.scan.completed，data.result_kind = healthy

存在 warning 或 critical：
- compliance.environment.scan.failed，data.result_kind = environment_unhealthy

雷达自身执行异常：
- compliance.environment.scan.failed，data.result_kind = execution_error，data.error = ErrorObject
```

`compliance.environment.scan.failed` 同时覆盖“环境健康未通过”和“雷达自身执行异常”两类失败。下游消费者不能只看事件 type，必须读取 `data.result_kind` 区分语义；当 `result_kind=execution_error` 时，`data.error` 必须使用 `00-基础契约.md` 定义的 `ErrorObject`。

事件 data 必须包含：

```text
scan_id
environment
result_kind：healthy | environment_unhealthy | execution_error
overall_status
affected_targets
critical_count
warning_count
report_artifact_uri
```

`result_kind=healthy` 表示雷达执行成功且所有目标健康，对应 `compliance.environment.scan.completed`。`result_kind=environment_unhealthy` 表示雷达执行成功，但环境存在 warning 或 critical。`result_kind=execution_error` 表示雷达自身执行失败，例如探测器异常、报告写入失败或事件发布失败。下游消费者必须优先读取 `result_kind`，不能只按事件 type 判断失败含义。

当前服务实现中，报告 artifact 写入失败会映射为 `ARTIFACT_WRITE_FAILED`；扫描结果事件发布失败会映射为 `EVENT_PUBLISH_FAILED`。服务会尽力发布 `result_kind=execution_error` 的失败事件，并在发布后继续向调用方抛出执行异常，避免调用链误判扫描成功。

## 12. 与 Agent 的关系

### 12.1 Pipeline Agent

Pipeline Agent 在构建执行前调用雷达前置预检。若 GitLab、Tekton、ArgoCD、K8s 中当前流程依赖项出现 `critical`，Pipeline Agent 必须终止真实构建或部署，避免无效执行浪费资源。

### 12.2 Diagnosis Agent

故障发生时，Diagnosis Agent 自动读取最近一次雷达报告，用于快速排除“依赖系统本身不可用”的基础问题，缩小根因排查范围。

### 12.3 Collaboration Agent

Collaboration Agent 订阅 `compliance.environment.scan.failed`。雷达发现 warning 或 critical 后，协同链路可以在业务故障发生前通知责任人，实现提前预警。

### 12.4 运维运营

日报生成器可纳入环境健康度数据，例如当日扫描次数、warning 次数、critical 次数、平均恢复时间和连续异常时长，用于评估平台稳定性。

## 13. 处理流程

```text
1. 接收定时、手动、前置预检或事件触发请求
2. 加载 environment_profile 和 radar_policy_config
3. 根据 mode 选择需要执行的探测器和检测层级
4. 对 GitLab、Tekton、ArgoCD、K8s 执行探测
5. 汇总每个 target 的状态、延迟、错误码、影响范围和建议
6. 生成 JSON 报告和 Markdown 报告
7. 写入 PostgreSQL 历史记录
8. 发布 compliance.environment.scan.completed 或 failed 事件
```

## 16. 验证方式

```text
定向验证：缺少 GitLab token 时 GitLab target 必须 critical。
定向验证：K8s 只读权限不足时必须记录 forbidden 错误。
定向验证：Token 7 天内过期时必须输出 warning。
定向验证：所有 target healthy 时输出 compliance.environment.scan.completed。
定向验证：任一 target warning 或 critical 时输出 compliance.environment.scan.failed。
定向验证：雷达自身执行异常时输出 compliance.environment.scan.failed，data.result_kind 必须为 execution_error，data.error 必须是 ErrorObject。
定向验证：Pipeline preflight 模式遇到 critical 必须返回阻断结果。
定向验证：报告 artifact 必须同时包含 JSON 和 Markdown。
```

以上命令只定义验证方式，本文档不要求现在运行。

## 17. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 某个外部系统凭据未就绪 | target 标记 critical，并在报告中写明 missing_credentials |
| 某个 API 不可用 | 输出 failed 事件，阻断依赖该系统的真实流程 |
| 权限过大 | 输出 warning，标记 excessive_permission |
| PostgreSQL 暂未部署 | 使用历史仓储接口写 artifact，但历史记录结构不得变化 |
| 事件总线不可用 | 保留本地 report artifact，并记录 EVENT_PUBLISH_FAILED |
| 探测器误报 | 允许人工标记 suppress，但必须记录原因和过期时间 |

## 18. 验收补充

```text
GitLab、Tekton、ArgoCD、K8s API 全部连通时，雷达输出 healthy 报告并发布 completed 事件。
任一目标出现 warning 或 critical 时，雷达输出 failed 事件并保留报告 artifact。
```

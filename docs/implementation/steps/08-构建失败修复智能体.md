# 08-构建失败修复智能体

## 1. 目标

在 GitLab CI 或 Tekton 构建、测试失败时，使用 FDE Agent Runtime 的 `code_runtime` 分析日志和相关文件，生成修复建议或补丁。所有修复必须进入 MR，不允许直接合入。

## 2. 不做什么

```text
不自动推送受保护分支
不自动合并 MR
不处理生产事故回滚
不修复超出明确日志证据的问题
```

## 3. 上游输入

```text
tekton.taskrun.completed(status=failed)
gitlab.pipeline.completed(status=failed)
build log artifact
test report artifact
源码 checkout
```

## 4. 输出结果

```text
build.fix.suggested
build-fix-report.md
build-fix.patch（可选）
修复分支或 MR 建议（可选）
```

## 5. 接口契约

### 5.1 输入

```json
{
  "failed_stage": "build-image",
  "build_log_uri": "artifacts/runs/corr-001/build.log",
  "test_report_uri": "artifacts/runs/corr-001/test.json",
  "commit_sha": "abc123",
  "application": "api-gateway"
}
```

### 5.2 输出

```json
{
  "status": "suggested",
  "failure_category": "dependency",
  "summary": "构建缺少 curl 依赖",
  "recommendation": "在 Dockerfile builder 阶段安装 curl",
  "patch_artifact_uri": "artifacts/runs/corr-001/build-fix.patch",
  "requires_human_review": true
}
```

## 6. 处理流程

```text
1. 消费失败事件
2. 拉取失败日志和相关文件
3. Agent Runtime 使用 `code_runtime`
4. 生成原因摘要和可选 patch
5. 保存 report 和 patch artifact
6. 需要时创建修复 MR 或提示人工创建
```

## 7. 文件结构

```text
agents/
  pipeline/
    build-fix/
      build-fix.md
ci/
  gitlab/
    build-fix.yml
prompts/
  pipeline/
    build-fix.md
schemas/
  agents/
    build-fix-result.schema.json
```

## 8. 与下一步衔接

如果修复建议仍未解决，异常进入 `09-诊断上下文构建器.md`，作为构建失败证据参与 Diagnosis 三层漏斗。

## 9. 验证方式

```text
定向验证：给定依赖缺失日志，应输出 dependency 分类。
定向验证：patch 存在时必须标记 requires_human_review=true。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 日志不足 | 输出需要人工补充上下文 |
| patch 风险高 | 只输出报告，不生成 patch |
| code_runtime 不可用 | 直接进入 Diagnosis Agent 处理 |

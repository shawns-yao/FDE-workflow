# 10-诊断规则引擎

## 1. 目标

实现 Diagnosis 三层漏斗的第一层：规则引擎。覆盖 Kubernetes 和部署链路的高频故障，快速返回标准根因和修复建议。规则命中时不调用 Claude API。

## 2. 不做什么

```text
不调用大模型
不做向量检索
不处理复杂跨系统推理
不输出没有 evidence 的结论
```

## 3. 上游输入

```text
diagnosis.context.built
```

## 4. 输出结果

```text
diagnosis.completed（规则命中）
diagnosis.rule.missed（规则未命中）
rule-match-report.json
```

## 5. 接口契约

### 5.1 输入

```json
{
  "context_id": "ctx-001",
  "signals": {
    "pod_status": "ImagePullBackOff",
    "event_reasons": ["Failed", "BackOff"]
  },
  "evidence_refs": []
}
```

### 5.2 规则命中输出

```json
{
  "diagnosis_id": "diag-001",
  "source": "rule",
  "funnel_layer": "rule",
  "category": "config",
  "severity": "high",
  "summary": "镜像拉取失败",
  "root_cause": "镜像地址或拉取凭据错误",
  "recommendation": "检查 image tag、registry secret 和镜像仓库权限",
  "confidence": 0.95,
  "evidence_refs": []
}
```

## 6. 处理流程

```text
1. 读取 diagnosis context
2. 按优先级匹配 Top 故障规则
3. 生成标准诊断结果
4. 命中则输出 diagnosis.completed
5. 未命中则输出 diagnosis.rule.missed
```

`source` 与 `funnel_layer` 的含义不同。`source` 表示诊断结论来源，必须对齐 `fde-platform-m1-contracts.md`；`funnel_layer` 表示三层漏斗位置。规则命中时固定为 `source=rule`、`funnel_layer=rule`。

## 7. 文件结构

```text
agents/
  diagnosis/
    rules/
      rule-engine.md
      top-failure-rules.md
schemas/
  diagnosis/
    rule-match-result.schema.json
```

## 8. 与下一步衔接

规则未命中时进入 `11-诊断知识库匹配.md`。规则命中时直接进入 `13-协同通知路由.md`。

## 9. 验证方式

```text
定向验证：ImagePullBackOff 样例命中镜像拉取规则。
定向验证：未知错误输出 diagnosis.rule.missed。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 规则误命中 | 置信度低于阈值时进入知识库匹配 |
| 规则覆盖不足 | 未命中不失败，进入下一层 |
| evidence 不足 | 输出 context_incomplete |

# 13-协同通知路由

## 1. 目标

实现 Collaboration Agent 的第一段能力：消费 `diagnosis.completed`，根据诊断结果、服务归属、环境和严重级别确定通知对象，并通过飞书连接器发送可追踪的通知。

## 2. 不做什么

```text
不直接诊断故障
不直接修改代码、配置或集群
不绕过飞书连接器发送消息
不把通知对象硬编码在业务逻辑中
```

## 3. 上游输入

```text
diagnosis.completed
service_ownership
gitlab_mr_metadata（可选）
deployment_metadata
environment_policy
```

## 4. 输出结果

```text
collaboration.notification.requested
collaboration.notification.sent
collaboration.notification.failed
notification-card.json
notification-routing-record.json
```

## 5. 接口契约

### 5.1 通知请求

```json
{
  "notification_id": "ntf-001",
  "diagnosis_id": "diag-002",
  "service": "payment-api",
  "environment": "test",
  "severity": "high",
  "target": {
    "type": "user",
    "id": "ou_xxx",
    "reason": "recent_committer"
  },
  "fallback_target": {
    "type": "group",
    "id": "oc_xxx"
  },
  "card": {
    "title": "payment-api 部署异常",
    "summary": "新版本启动后持续 CrashLoopBackOff",
    "actions": ["acknowledge", "claim", "mark_fixed"]
  }
}
```

### 5.2 通知结果

```json
{
  "notification_id": "ntf-001",
  "status": "sent",
  "channel": "feishu",
  "message_id": "om_xxx",
  "target": {
    "type": "user",
    "id": "ou_xxx"
  },
  "artifact_uri": "artifacts/collaboration/ntf-001/notification-card.json"
}
```

## 6. 处理流程

```text
1. 读取 diagnosis.completed 和部署元数据
2. 解析服务归属配置，确定主通知对象和兜底对象
3. 严重级别为 high 或 critical 时增加运维兜底群
4. 通过 Agent Runtime 调用 Claude API 生成通知摘要
5. 校验 notification-card schema
6. 调用飞书连接器发送卡片
7. 记录 notification-routing-record
8. 输出 collaboration.notification.sent 或 collaboration.notification.failed
```

## 7. 文件结构

```text
agents/
  collaboration/
    notification-router/
      route-notification.md
connectors/
  feishu/
    send-card.md
schemas/
  collaboration/
    notification-request.schema.json
    notification-result.schema.json
    notification-card.schema.json
configs/
  ownership/
    service-ownership.example.yaml
```

## 8. 与下一步衔接

发送成功后进入 `14-协同进度追踪.md`。下游依赖 `notification_id`、飞书 `message_id`、卡片按钮回调标识和通知目标，不能只保存一段普通文本。

## 9. 验证方式

```text
定向验证：用 mock diagnosis.completed 生成 notification-request。
定向验证：服务归属缺失时必须进入 fallback_target。
定向验证：飞书发送失败时输出 collaboration.notification.failed。
定向验证：通知卡片必须通过 schema 校验。
```

## 10. 风险与降级

| 风险 | 降级 |
| --- | --- |
| 服务归属不完整 | 发送到兜底运维群，并标记 owner_unknown |
| Claude API 不可用 | 使用规则模板生成通知摘要 |
| 飞书接口不可用 | 保存 notification-card artifact，发出 failed 事件等待重试 |
| prod 环境严重故障 | 同时通知开发责任人和运维值班群 |

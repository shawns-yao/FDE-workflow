# 面向FDE的workstations产品设计方案

**定位说明**：本文保留为长期产品远景与方向性设计，不作为 M1 实施依据。M1 的范围、边界、验收和接口契约以 `docs/architecture/fde-platform-m1-design.md`、`docs/architecture/fde-platform-m1-contracts.md` 和 `docs/requirements/milestone-1-plan.md` 为准。

## 一、机会判断：为什么现在做这件事

### 1.1 市场窗口

核心判断：FDE不是"又一个技术岗位"，而是AI应用落地时代的关键瓶颈角色。岗位爆发+人才稀缺+工具缺失，构成了一个明确的工具市场空白。

### 1.2 现有工具为什么不够用

当前AI工具生态高度集中在"如何构建AI应用"，但FDE的核心挑战是"如何在客户现场证明AI价值并推动落地"。这是两个完全不同的命题。

结论：现有工具解决的是"Build"问题，FDE需要的是"Deliver & Validate"工具。这是一个未被满足的蓝海市场。

### 3.2 竞品分层解析

第一层：通用AI开发平台（直接竞品，但错位）

关键洞察：这一层所有工具的共同问题是 —— 它们解决"Build"问题，不解决"Deliver"问题。FDE用这些工具做出Demo后，仍然要独自面对客户现场的所有非技术挑战。

第二层：企业级AI平台（间接竞品，模式对标）

关键洞察：Palantir和Nexus验证了一个核心假设 —— "平台+FDE"模式是AI应用落地的最佳路径。但它们都是国外产品，且面向大型企业级销售，中国市场的FDE个人/团队效率工具完全空白。

第三层：国内大厂Agent平台（生态竞品，场景不同）

关键洞察：国内大厂都在做"Agent开发平台"，但没有任何一家做"FDE交付平台"。这是生态位空白。

第四层：海外FDE专用工具（验证方向，未入华）

关键洞察：海外已有FDE专用工具萌芽，但全部未进入中国市场，且多为单点工具。系统级FDE Workstation在全球范围内也是首创。

### 3.3 竞品矩阵：功能维度对比

结论：

红色区域（我们独有）：客户现场环境适配、组织阻力诊断、业务价值量化、客户共创评估、FDE工作流SOP

黄色区域（我们有优势）：私有化/离线部署、行业模板市场、评估驱动开发、国产化适配

绿色区域（竞品更强）：通用可视化编排（Dify/Coze更成熟）、生态丰富度（大厂更强）

## 六、模块设计（Module Design）

### 6.1 Layer 1: 现场工具层（Field Toolkit）

设计目标：确保FDE在"任何客户现场"都能立即开始工作，不受环境限制。

模块 1.1: 离线开发环境（Offline-IDE）

解决的问题：客户内网无公网、无Docker、系统老旧

核心能力:

完全离线的AI应用原型开发环境，无需公网即可在客户内网运行

预置轻量级模型（Qwen3.5-8B、ChatGLM4-8B等），支持CPU或游戏卡推理

离线RAG引擎：基于FAISS的本地向量检索，支持PDF/Word/Excel解析

内网Git管理：本地Git服务器+版本管理

一键导出：导出为可离线运行的Docker镜像或单文件可执行包

技术规格:

运行环境: Docker / 单机可执行文件

模型支持: Qwen3.5-8B, ChatGLM4-8B, Llama-3-8B (INT4量化)

向量库: FAISS-CPU

文档解析: 本地OCR + 结构化提取

最小硬件: 8核CPU / 16GB内存 / 50GB存储

与竞品差异：Dify/Coze必须联网；LangChain需自建环境。我们是开箱即用的离线IDE。

模块 1.2: 合规检测雷达（Compliance Radar）

解决的问题：进入现场前不知道客户环境限制，导致部署失败

核心能力:

等保/密评预检：根据客户行业自动输出合规检查清单

网络拓扑探测：扫描可用端口、防火墙规则、代理配置

数据分级评估：识别数据敏感度，推荐部署策略（公有云/私有云/混合云/纯内网）

国产化适配清单：自动检测是否需要适配国产芯片（昇腾/寒武纪）、国产操作系统（麒麟/统信）

输出物:

📋 《客户现场部署可行性报告》

├── 网络环境: ✅ 内网可用 / ⚠️ 需申请白名单 / ❌ 完全隔离

├── 数据合规: 三级等保 → 推荐私有化部署

├── 国产化要求: 需适配昇腾910B

├── 预计部署时间: 2小时（标准环境）/ 8小时（隔离环境）

└── 风险项: 客户无Docker环境，需提前准备二进制包

与竞品差异：Palantir有合规能力但面向国外标准；国内无同类产品。

模块 1.3: 现场作战背包（Field-Pack）

解决的问题：现场高频工具分散，故障排查、客户培训、日报编写效率低

工具清单:

与竞品差异：Granola/Read.ai只做会议记录；我们是FDE全场景工具集合。

### 6.2 Layer 2: 客户洞察层（Customer Intelligence）

设计目标：将FDE从"被动响应需求"升级为"主动诊断客户、推动变革"。

模块 2.1: 组织阻力热力图（Org-Heatmap）

解决的问题：不知道客户内部谁支持、谁反对、为什么反对

核心能力:

利益相关者识别：自动识别关键角色（决策者、影响者、执行者、反对者）

阻力指数评估：为每个部门/角色计算"AI变革阻力指数"（0-100）

阻力根因分析：识别阻力来源（技能恐惧、利益受损、资源争夺、文化冲突）

破冰策略推荐：根据阻力类型推荐策略（如"培训赋能"应对技能恐惧，"利益再分配"应对利益受损）

可视化输出:

🏢 客户组织阻力热力图

├── 高层（CEO/CTO）: 🔥 阻力 15 → 积极推动型

├── 中层（部门经理）: 🔥🔥🔥 阻力 78 → 最大阻力源（根因: 利益受损恐惧）

│   └── 推荐策略: 让其成为AI项目"联合负责人"，纳入KPI

├── 基层（业务员工）: 🔥🔥 阻力 45 → 观望型（根因: 技能恐惧）

│   └── 推荐策略: 提供"AI辅助而非替代"培训，展示减负效果

└── IT部门: 🔥🔥 阻力 52 → 技术保守型（根因: 安全担忧）

    └── 推荐策略: 邀请参与安全评审，给予技术话语权

与竞品差异：全球首创。Palantir有数据平台但无组织变革诊断；Nexus有FDE但无此工具。

模块 2.2: 利益相关者图谱（Stake-Map）

解决的问题：项目关键人态度变化无法追踪，导致项目突然受阻

核心能力:

关系网络可视化：展示各利益相关者的影响力、态度、关系亲疏

态度追踪：记录每次沟通后的态度变化，预警"态度恶化"

沟通策略库：根据角色类型推荐沟通话术

关键人备忘录：记录个人诉求、顾虑点、决策偏好

预警机制:

⚠️ 预警: 财务部张总的态度从"支持"变为"中立"

触发条件: 上次会议提到"预算超支风险"

建议动作: 48小时内安排一对一沟通，提供ROI详细测算

与竞品差异：CRM工具有客户管理但无"组织内部政治图谱"功能。

模块 2.3: 异步访谈Agent（Async-Chat）

解决的问题：客户业务人员没时间配合访谈，需求收集不完整

核心能力:

智能访谈设计：根据行业自动生成访谈提纲

多轮追问：AI Agent进行对话式追问（"您提到审批慢，具体是哪个环节？需要多长时间？"）

痛点自动提取：从访谈记录中提取Top 10痛点，按频率和严重程度排序

场景映射：将痛点自动映射到可落地的AI场景

输出物:

📊 《客户痛点与场景映射报告》

├── Top 痛点:

│   1. 合同审批平均耗时7天（影响: 业务效率，提及率: 89%）

│   2. 客户投诉响应慢（影响: 客户满意度，提及率: 76%）

│   3. 报表制作占用2人/天（影响: 人力成本，提及率: 65%）

├── 可落地场景:

│   1. 合同审批 → 智能审批助手（预期提效60%）

│   2. 投诉响应 → 智能客服分流（预期响应时间降至2小时）

│   3. 报表制作 → 自然语言报表生成（预期节省80%时间）

└── 优先级建议: 先攻"合同审批"（高痛点+高可行性+影响部门少）

与竞品差异：Perspective AI做类似事情但仅海外可用；我们是集成在FDE工作流中的访谈工具。

### 6.3 Layer 3: 需求转化层（Requirement Translation）

设计目标：将客户模糊语言转化为精确技术方案，防止需求蔓延。

模块 3.1: 需求翻译器（Req-Trans）

解决的问题：客户说"想用AI提高效率"，FDE不知道具体做什么

核心能力:

业务语言解析：输入模糊需求，输出结构化拆解

技术可行性评估：自动评估可行性（高/中/低），标注风险点

Scope界定：明确MVP边界，防止需求蔓延

需求确认书生成：生成客户可签字的《AI需求确认书》

翻译示例:

输入: "我们想用AI做智能客服"

输出:

├── 业务需求拆解:

│   ├── 核心诉求: 降低人工客服工作量

│   ├── 覆盖渠道: 微信/网页/APP（需确认）

│   ├── 知识范围: 产品咨询/售后/投诉（需确认）

│   └── 接入系统: 现有客服系统（需API文档）

├── 技术方案映射:

│   ├── 技术栈: RAG + 大模型对话

│   ├── 数据源: 产品手册 + FAQ + 历史对话记录

│   ├── 模型选择: 私有化部署Qwen-14B（数据敏感）

│   └── 集成方式: 现有客服系统Webhook接入

├── MVP Scope:

│   ✅ 第一版: 产品咨询自动回复（覆盖80%常见问题）

│   ❌ 第二版: 售后工单自动创建

│   ❌ 第三版: 情绪识别与升级预警

├── 风险点:

│   ⚠️ 客户无历史对话数据，需2周数据收集期

│   ⚠️ 现有客服系统API文档缺失，需客户IT配合

└── 预计周期: 2周PoC + 4周生产部署

与竞品差异：Dify/Coze假设用户知道要做什么；我们帮助用户搞清楚要做什么。

模块 3.2: 场景挖掘Agent（Scene-Miner）

解决的问题：客户"答不上来具体想做什么"，项目无法启动

核心能力:

行业场景库：预置金融、制造、医疗、零售、政务等Top 50高频AI场景

流程扫描：根据客户SOP文档自动识别"高AI适配度环节"

对标案例推荐：推荐同行业已落地案例（脱敏后）

场景可行性打分：从"技术成熟度、数据就绪度、组织接受度、ROI清晰度"四维度打分

与竞品差异：首创"帮客户找需求"功能。所有竞品都假设需求已明确。

模块 3.3: 价值计算器（ROI-Calc）

解决的问题：FDE无法向客户高管证明AI项目的财务价值

核心能力:

投入测算：自动计算全周期成本（软件、硬件、人力、运维）

收益量化：将"效率提升"转化为"节省人力成本/增加收入/降低风险"的财务数字

ROI模型：生成3年ROI曲线，标注盈亏平衡点

敏感性分析：展示"如果效率只提升30%而非50%，ROI如何变化"

高管汇报PPT：一键生成一页纸Executive Summary

与竞品差异：Palantir有数据但无面向FDE的ROI工具；我们是FDE向高管汇报的专用工具。

### 6.4 Layer 4: 原型工厂层（Prototype Factory）

设计目标：将FDE从"写代码"变为"配模板"，实现规模化交付。

模块 4.1: 行业模板市场（Template Hub）

解决的问题：每个项目从零开始，FDE被重复性定制压垮

模板体系:

Template Hub 结构:

├── 金融行业

│   ├── 智能信贷审批模板（含风控规则引擎、征信对接、审批流）

│   ├── 监管报表自动生成模板（含1104/人行报表映射）

│   ├── 智能客服模板（含寒暄/业务/投诉多轮对话）

│   └── 投研摘要模板（含研报解析、舆情监控、观点提取）

├── 制造行业

│   ├── 设备预测性维护模板（含传感器数据接入、异常检测、工单生成）

│   ├── 质量检测视觉模板（含缺陷分类、置信度阈值、MES对接）

│   └── 供应链智能排产模板

├── 医疗行业

│   ├── 病历结构化模板（含ICD编码、实体识别、时间轴提取）

│   ├── 智能导诊模板（含症状分诊、科室推荐、预约对接）

│   └── 医学文献问答模板

├── 政务行业

│   ├── 12345热线智能分派模板

│   ├── 政策智能问答模板

│   └── 公文自动生成模板

└── 通用模板

    ├── RAG知识库模板（多文档类型、权限分级、溯源）

    ├── AI Agent编排模板（多Agent协作、工具调用、记忆管理）

    └── 数据可视化模板（自然语言查询、自动图表生成）

模板使用SOP:

选择模板：根据行业+场景选择基础模板

配置参数：填写客户专属参数（数据源路径、API地址、业务规则阈值）

注入数据：上传客户样本数据（自动脱敏），进行适配

微调验证：在模板预设的评估集上验证效果

一键导出：导出为可部署包（Docker/二进制/Edge Box镜像）

与竞品差异：Coze模板丰富但面向C端/轻量场景；Dify有社区模板但无行业深度。我们聚焦企业级B端行业模板。

模块 4.2: 私有化部署盒（Edge Box）

解决的问题：客户现场系统老旧、网络隔离、部署复杂

产品形态:

Edge Box Mini（演示版）:

├── 硬件: ARM架构边缘计算盒（15cm×10cm×3cm）

├── 算力: 8TOPS NPU，支持INT8推理

├── 预装: FDE Workstation Runtime + 选定模板 + 轻量模型

├── 接口: HDMI、USB、网口

├── 供电: 标准Type-C，即插即用

└── 场景: 客户现场PoC演示，无需客户IT配合

Edge Box Pro（生产版）:

├── 硬件: x86服务器/国产信创服务器（昇腾/寒武纪）

├── 算力: 按需配置（支持多卡并行）

├── 预装: 完整FDE Runtime + 模型仓库 + 监控套件

├── 部署: 支持K8s/裸金属/虚拟机

└── 场景: 生产环境私有化部署

部署流程:

FDE携带Edge Box Mini → 客户会议室插电 → 自动连接内网/离线运行

→ 加载客户样本数据 → 15分钟完成Demo环境搭建

→ 向客户高管现场演示 → 获得认可后 → 无缝迁移至Edge Box Pro生产环境

与竞品差异：首创"硬件+软件"一体化交付。Dify/Coze/LangChain均无硬件产品。

模块 4.3: 数据治理工坊（Data Forge）

解决的问题：企业数据质量差（准确性、完整性、一致性、可用性），FDE花50%时间清理数据

核心能力:

数据质量诊断：自动扫描五大质量问题

智能清洗流水线：去重、纠错、格式标准化、缺失值填充

知识库构建向导：非结构化文档→结构化知识库

矛盾检测：自动识别知识库中的矛盾信息

数据脱敏：一键脱敏，生成可演示的脱敏数据集

与竞品差异：Dify有基础RAG数据管理但无质量诊断+矛盾检测+脱敏一体化工具。

### 6.5 Layer 5: 交付验证层（Delivery & Validation）

设计目标：跨越"PoC到生产的死亡之谷"，确保交付即产生价值。

模块 5.1: 评估驱动引擎（EDD Core）

解决的问题：AI项目缺乏评估体系，Demo无法证明价值，客户不验收

核心能力:

评估指标体系：技术指标（准确率、召回率、延迟）+ 业务指标（效率提升率、用户满意度、错误率下降）

Golden Dataset管理：创建、管理、版本控制客户认可的"标准测试集"

A/B测试框架：支持新旧方案对比，自动计算统计显著性

回归测试：每次更新后自动运行全量评估，防止退化

评估报告生成：自动生成客户可读的评估报告

EDD工作流:

Step 1: 定义成功标准 → "审批准确率需≥95%，平均处理时间≤2小时"

Step 2: 构建Golden Dataset → 客户业务专家标注100条标准测试样本

Step 3: 基线评估 → 记录当前人工处理的基准数据

Step 4: 迭代开发 → 每完成一个功能点，立即在Golden Dataset上评估

Step 5: 对比验证 → 新方案 vs 基线 vs 旧方案，三向对比

Step 6: 客户验收 → 客户业务专家参与评估，签字确认

Step 7: 持续监控 → 生产环境实时监控，偏离阈值自动告警

与竞品差异：LangSmith有技术监控但无业务级评估+客户参与验收；我们是评估驱动开发方法论的工具化。

模块 5.2: 客户共创看板（Co-Eval Dashboard）

解决的问题：客户业务人员无法参与AI评估，验收标准由技术人员单方面定义

核心能力:

业务语言评估界面：客户看到的是"这个回答解决了我的问题吗？（是/否/部分）"，而非"BLEU Score = 0.72"

反馈即时转化：客户的"这个回答不对"自动转化为技术团队的测试用例和优化任务

多轮评估追踪：记录同一问题在V1/V2/V3版本的回答质量变化曲线

满意度热力图：按业务场景展示满意度分布，快速定位"最不满意场景"

高管驾驶舱：一页展示项目进度、关键指标达成率、客户满意度、风险预警

与竞品差异：全球首创"客户可参与的共创评估平台"。所有竞品都是技术人员内部工具。

模块 5.3: 生产迁移工厂（Prod-Move Factory）

解决的问题：PoC验证通过后，迁移到生产环境困难、风险高、无回滚能力

核心能力:

环境差异检测：自动对比PoC与生产环境差异

灰度发布：支持1%→10%→50%→100%渐进式发布

回滚机制：一键回滚到上一稳定版本

监控埋点：自动注入业务监控埋点

运维手册生成：自动生成客户IT部门可执行的运维手册和应急预案

与竞品差异：MLOps平台（如Kubeflow）面向技术团队；我们是面向FDE+客户IT的轻量级迁移工具。
